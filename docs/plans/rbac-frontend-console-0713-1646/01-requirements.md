# RBAC 配套前端需求分析

> 视角：requirements-analyst。本文只描述基于当前工作树可确认的需求、规则和边界；未在代码中落地的能力均标为“计划新增”或“待验证”。

## 1. 背景与用户问题

当前 `capability-showcase-frontend` 是 Vue 3 + TypeScript + Vite 的独立静态 SPA。它已经有账号密码登录、内存访问令牌、httpOnly 刷新 cookie、路由守卫、API Key 兼容入口、能力目录和 RAG 工作台。当前分支 `feat/rbac-shared-kb` 正在把账号体系扩展为角色到 scope 的 RBAC，并把共享知识库实现为保留租户 `__public__`。

前端需要把后端“角色决定能做什么、租户决定能看到什么、共享库是所有租户可读的额外分区”转化为可理解、可操作且不误导的界面，而不是只增加几个表单。

核心使用者分为：

- 普通用户：登录、理解当前租户与权限、只看到或明确识别自己能做的操作。
- 知识库编辑者：向自己的租户库入库；拥有 `public-ingest` 时可选择共享入库。
- 平台管理员：用 `role-admin` 管理全局角色、用户、租户归属和启用状态。
- 演示/开发用户：仍可使用现有 demo 账号和可选 API Key，但必须清楚当前真正生效的凭证。

## 2. 已确认业务规则

以下规则均能在当前代码中找到直接依据：

1. 账号登录和刷新返回 `UserView(username, tenant, scopes)`；`scopes` 是签发时的有效 scopes，不包含 roles。见 `auth-service/.../dto/UserView.java`、`AuthService.issueFor(...)`。
2. 有效 scopes 在 RBAC 开启时等于用户直配 scopes 与角色展开 scopes 的并集；关闭时只使用直配 scopes。见 `AuthService.issueFor(...)`、`RoleService.effectiveScopes(...)`。
3. 角色是全局定义，不按租户隔离；用户只有一个 tenant。见 `Role.java`、`UserAccount.java`、`AdminService` 注释与实现。
4. `/auth/admin/**` 是平台级管理接口，每个端点都要求 `role-admin`。前端隐藏入口只能改善体验，不能替代服务端 403。
5. 角色/权限变化不会回溯已签发的 access token；登录或刷新才重新展开。当前 access token 默认 60 分钟，见 auth/edge 的 `SESSION_ACCESS_TTL`。
6. 自助注册默认关闭，且当前代码要求 RBAC 与 registration 两个开关同时开启；成功注册直接返回登录会话。
7. 注册映射通过用户名邮箱域选择 tenant + roles，未命中时使用默认 tenant/default role；真实用户 tenant 不允许是 `__public__`。
8. 共享知识库不是新物理系统，而是 `tenantId=__public__` 的保留分区。开启 `app.rag.public.enabled` 后，查询会把当前租户和共享分区结果合并。
9. 普通租户入库要求 `ingest`；共享入库要求 `public-ingest`。上传的 `visibility=public|shared` 都被视为共享。
10. 当前共享图片入库明确不支持；文件和 JSON 文本入库支持共享。
11. 当前文档列表、详情、删除只作用于调用者 tenant；没有共享库列表/详情/删除入口。
12. 当前查询响应 `KnowledgeQueryService.Hit` 没有 tenant/visibility 字段，前端无法可靠给命中打“共享”标记。
13. 登录会话与手输 API Key 并存，且 API Key 非空时覆盖 Bearer。登录用户 scopes 因此不一定代表当前能力请求的真实权限。
14. access token 和 API Key 都只存内存；刷新 cookie 为 httpOnly。该安全边界必须保持。

## 3. 功能需求

### 3.1 身份与会话体验

- 登录后在顶栏明确显示 username、tenant、当前凭证模式和有效权限摘要。
- Bearer 模式可依据 `AuthUser.scopes` 预判操作权限；API Key 覆盖模式必须显示“权限未知，以服务端结果为准”，不能拿登录用户 scopes 误判。
- 401 时沿用现有单飞 refresh 和一次重试；refresh 失败回登录页并保留安全的站内 redirect。
- 403 不应再统一提示“API Key 缺 scope”；要根据当前凭证模式提示“当前账号”或“当前 API Key”。
- 管理员修改自己的 tenant/roles/enabled/password 时必须有显著风险提示；修改后按后端最终会话策略退出或重新刷新，不能继续展示旧权限为最新状态。

### 3.2 登录与注册

- 登录表单保留现有校验、密码显隐、错误人话化和深链返回。
- 当服务端公开配置确认注册开启时，显示“创建账号”入口；关闭时不展示或给出明确关闭状态。
- 注册页只收 username/password/confirmPassword；tenant 和 roles 由服务端规则决定，前端不得让用户自行选择。
- 密码长度提示必须来自服务端公开配置或部署配置，不能复制一个可能漂移的常量。
- 当前登录页把 `demo12345` 编进静态包并一键提交。生产方案必须默认关闭该行为；演示卡可改为只填 username，或只在明确的 demo 构建中启用。

### 3.3 权限感知的能力控制台

- Bearer 模式下，能力目录的 `requiredScopes` 与当前有效 scopes 做前置比对：缺权时允许查看说明和 curl，但禁用执行并说明缺少的 scope。
- API Key 模式下保持反应式鉴权，因为 Key 对前端是不透明的；不得错误禁用。
- 侧栏、命令面板、收藏、深链和专用工作台必须使用同一权限判定函数，避免一个入口隐藏、另一个入口仍可点。
- “不可见”和“不可执行”需要区分：默认保留能力可见性以方便发现和申请权限；破坏性能力仍沿用二次确认。
- 直接访问无权限管理路由时显示专用 403 页面，而不是空白、404 或循环跳转。

### 3.4 用户管理

- 管理员可按 username、tenant、role、enabled 检索和分页浏览用户。
- 列表清楚区分 tenant、roles、direct scopes、effective scopes、enabled；当前 API 只返回名为 `scopes` 的 direct scopes，不能在 UI 中误标为 effective scopes。
- 创建用户：username、初始密码、tenant、roles、enabled；角色从真实角色列表选择。
- 编辑用户：tenant、roles、enabled、可选重置密码。空密码必须表示“不修改”，不能被序列化成清空密码。
- 启停、删除、移出租户/管理员角色均需二次确认和影响说明。
- 并发编辑必须检测陈旧版本并让用户刷新/合并，不能静默 last-write-wins 覆盖另一管理员操作。
- 防止删除/禁用最后一个有效平台管理员必须由后端保证，前端同时给出可读错误。

### 3.5 角色与权限管理

- 展示角色名、说明、scopes、绑定用户数和版本。
- 创建角色与编辑角色使用分组 scope 选择器，并允许显示当前代码尚未认识的历史 scope，避免读取旧数据时丢失。
- 修改角色 scopes 前展示受影响用户数；删除被引用角色时明确阻止并提供查看绑定用户入口。
- 角色名创建后默认不可改名；当前角色主键就是 name，改名实质是建新、迁移关系、删旧，不应伪装成普通编辑。
- scope 是字符串权限标识，不在本期提供 scope CRUD 或角色继承。

### 3.6 租户与共享知识库

- RAG 文档区改为“我的租户库 / 共享知识库”双视图，始终展示当前 tenant。
- 入库时明确选择“当前租户”或“共享”；共享选项仅在 Bearer 有 `public-ingest` 时直接开放。API Key 模式可开放但标记权限未知。
- 共享入库需确认“所有租户检索均可能命中”；共享图片选项禁用并解释后端不支持。
- 文档行显示可见性、上传时间、版本、类目、切片数和大小；共享删除必须单独确认。
- 查询命中显示“租户库/共享库”来源。该体验依赖后端增加可靠 visibility 字段，不能根据 docId 或 displayName 猜测。
- 共享读开关关闭时，UI 不得声称查询包含共享库；需要运行时能力信息或显式 kill switch。

## 4. 体验与性能要求

- 管理中心使用独立路由和懒加载 chunk；普通用户不下载完整管理页面代码。
- 管理页不应被静态 capability catalog 加载失败阻塞。当前 `App.vue` 对所有非公开路由施加目录门禁，需要拆分。
- 用户/角色列表使用服务端分页、筛选、防抖和请求取消；切换筛选时不闪回旧结果。
- 表单提交期间禁用重复提交；创建请求网络结果不明时先查目标是否已存在，不自动重放含密码的请求。
- 成功反馈就地呈现，失败保留用户输入；弹窗关闭后焦点返回触发按钮。
- 桌面优先但支持 768px 以上平板；关键流程必须纯键盘可操作，颜色不是唯一状态表达。
- 页面不使用 `v-html` 渲染角色说明、用户名、tenant 或服务端错误文本。

建议验收预算：

- 已登录后切换管理子页的本地交互响应小于 100ms（不含网络）。
- 50 行列表 API 成功返回后 200ms 内完成稳定渲染。
- 4G 模拟下管理路由懒加载 + 首屏数据 p75 小于 2.5s；不把整个能力目录作为前置依赖。
- 列表筛选请求防抖 250–350ms；同类在途请求最多一个。

## 5. 非目标

- 不在前端实现真正授权；服务端仍是唯一安全边界。
- 不做 tenant-scoped role、角色继承、ABAC、资源级 ACL 或审批流。
- 不让用户在注册页选择 tenant/role，也不在浏览器解释邮箱域映射规则。
- 不在本期提供 scope 定义的运行时 CRUD。
- 不把 access token、角色缓存或管理员草稿写入 localStorage/sessionStorage。
- 不为共享知识库复制一套独立向量库/ES 入库链。
- 不引入实时 RBAC 消息推送；角色变更后的会话一致性先按刷新/过期规则处理。
- 不在本规划任务修改任何业务代码。

## 6. 边界条件与失败场景

- 登录后角色被管理员删除/降权：页面可能仍持旧 access token；每个 403 都要以服务端为准，并提供重新登录动作。
- 两位管理员同时编辑同一用户/角色：后提交者固定收到412 `precondition_failed` 后展示差异和刷新按钮；409只表示重复名、角色被引用、最后管理员保护等业务冲突。
- 管理员删除自己、禁用自己或移除自己的 `role-admin`：服务端必须保护最后管理员；非最后管理员自改成功后前端应退出管理态。
- 用户列表加载成功、角色列表失败：用户页保持可读，但禁用创建/角色修改，不能把空角色列表当“无角色”。
- 网络超时但创建可能已提交：按 username/role name 查询后对账，不自动用密码重试。
- API Key 覆盖登录：顶栏和所有表单必须显示覆盖状态；管理中心默认只用 Bearer，避免用未知 Key 意外做平台级操作。
- refresh 多请求并发：复用现有 single-flight；只允许每个原请求重试一次。
- `RAG_PUBLIC_ENABLED=false` 但用户有 `public-ingest`：权限存在不代表功能开启，UI 需同时考虑能力开关。
- 私有与共享文档同名：docId 因 tenantId 不同而不同；UI 必须同时显示 visibility，不能只用名称作为 key。
- 同名文档重传是版本替换流程，当前 `DocumentService.upload(...)` 会先删旧数据再写新数据；失败可能产生跨 sink 不一致，前端应显示结果而不能宣称原子完成。
- 共享图片、缺少文件、未知角色、保留 tenant、重复 username、账号禁用、注册关闭和限流均需保留服务端错误语义。

## 7. 歧义与待验证项

以下事项不能从当前代码得出最终产品决策：

1. **待验证：**所有登录用户是否允许列出共享文档元数据；本计划默认“共享可读即元数据可读”，写/删仍需 `public-ingest`。
2. **待验证：**是否允许查看共享文档完整正文。当前 `DocumentInfo` 只有元数据，`GET /rag/documents/{docId}` 也不返回正文。
3. **待验证：**生产是否允许保留 API Key 管理 RBAC。推荐管理中心只使用 Bearer；服务到服务 Key 是否保留 `role-admin` 需安全负责人确认。
4. **待验证：**默认注册 tenant `public` 是否为真实业务租户，不能与保留分区 `__public__` 混淆。
5. **待验证：**角色 direct scopes 是否继续对管理员可编辑。当前 Admin DTO 不接收 direct scopes，但用户模型仍保留该字段。
6. **待验证：**共享知识库开关和注册开关是否需要运行时公开查询接口；推荐新增最小只读配置接口，而不是只靠 Vite 构建变量。
7. **待验证：**是否需要多语言；当前控制台以中文为主，本期默认中文。

## 8. 验收标准

- 普通用户、知识库编辑者、共享库管理员、平台管理员四类账号看到的入口和可执行状态符合其有效 scopes。
- 直接深链不能绕过 UI 路由权限提示，且服务端 403 始终保留。
- 管理员能完成用户创建、编辑、角色替换、启停、删除，以及角色创建、scope 编辑、删除受阻的完整闭环。
- 并发编辑不会静默覆盖；陈旧写有明确冲突界面。
- 注册开关关闭/开启、密码策略和 409/429 等状态在人机交互上正确呈现。
- RAG 可选择租户/共享入库；共享权限和图片限制正确；列表与查询命中能可靠标识 visibility。
- 角色/租户变更后的旧会话行为被明确提示和测试，不把前端缓存当权威。
- API Key 覆盖状态清晰，管理中心不会无意使用覆盖 Key。
- 所有敏感凭证仍只在内存/httpOnly cookie 中；生产构建不包含硬编码 demo 密码。
- 单元、组件、接口契约、E2E、跨租户隔离、并发和回滚测试全部满足 `test-plan.md`。
