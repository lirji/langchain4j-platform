## 0. 一页纸结论（给主控与各专业 Agent）

本控制台不是"再堆一个 CRUD 后台"，而是把一个**已经很在意"诚实呈现能力状态"的能力试用台**，扩展出一个**Bearer-only、领域隔离、带并发安全护栏的管理中心**，同时给能力域补上**"我现在用哪个身份、能做什么"**的确定性。三条产品主线，按此优先级：

1. **身份确定性（贯穿全站，P0 且安全关键）**：任何时刻用户都能一眼看清 `用户名 · 租户 · 凭证模式（Bearer / API Key / 无）`；当 API Key 覆盖登录身份时给**高对比警告 + 一键清除**，且**管理入口在此状态下消失**（管理只认 Bearer）。这是"顺手/不困惑"的地基，也是防越权误操作的第一道。
2. **管理闭环（P0，价值核心）**：管理员登录 → 进管理中心 → 检索用户 → 改角色/启停/重置密码 → 保存；建角色 → 选 scope → 看影响用户数 → 保存；删在用角色被挡。**不可静默覆盖**是不可谈判的红线：版本冲突必须有清晰可恢复、不丢草稿的路径。
3. **注册自助 + 共享知识库（P1，可与管理闭环并行切片）**：注册页运行时遵从后端开关与密码策略、不让用户选 tenant/role；RAG 工作台升级为"当前租户库 / 共享库"双视图，命中带"租户/共享"来源标识。

一句话产品判断：**管理员要的是"快"，但更要的是"敢按保存"**。所有设计取舍围绕"降低犹豫成本"——预判影响、显式确认、冲突可恢复、缺权说清原因。

## 1. 用户画像与场景

### 1.1 平台管理员（`role-admin`）— 主画像

| 维度 | 描述 |
|---|---|
| 谁 | 平台/运维负责人（对应现有 demo 的 `alice`，租户 `acme`，全权限）。中等技术水平，懂 scope 概念但不想背 scope 字符串。 |
| 目标 | ①给新同事开账号并授权；②出安全事件时**秒级**停用账号/降权；③重置忘密码用户；④设计/调整角色的 scope 组合；⑤在按下保存前，**看清这一步会影响谁**。 |
| 使用频率 | **突发式**：入职潮/项目启动时批量操作，平时低频；事件响应时要求"最快路径"。 |
| 关键痛点 | ①看不到用户的**有效 scopes**，不确定授权是否到位；②怕两人同时改一个用户/角色**静默互相覆盖**；③怕**把自己锁死**（删掉最后一个管理员/移除自己 role-admin）；④删角色时怕**悄悄破坏**一批人的权限；⑤操作后**不确定是否真的生效**（access token 延迟问题）。 |
| 成功标准 | "改角色到保存 ≤ 4 步"；"任何破坏性操作前都有一次可理解的确认或影响预览"；"冲突了不丢我刚填的东西"。 |

### 1.2 普通租户用户 — 高频画像

| 维度 | 描述 |
|---|---|
| 谁 | 业务侧使用者（对应 `bob`=只读对话/globex、`analyst-a`=对话+数据分析/tenantA）。只关心"我能用的能力"。 |
| 目标 | ①用自己有权的能力（chat/rag/agent…）；②往**本租户**知识库上传/检索文档；③（共享开时）浏览共享库；④尽快开始，不被后台噪音干扰。 |
| 使用频率 | **日常高频**。 |
| 关键痛点 | ①某能力锁着，分不清是"没权限(scope)"还是"没开(flag)"；②搞不清当前生效的是登录身份还是手填的 API Key；③点了一个深链结果 Forbidden，不知为何。 |
| 成功标准 | "管理相关的一切对我不可见、不占地方"；"能力为什么不能用，一句话说清且可发现"。 |

### 1.3 新注册者 — 一次性画像

| 维度 | 描述 |
|---|---|
| 谁 | 首次拿到平台地址的新人，可能非技术。 |
| 目标 | 尽快拿到账号并开始用。 |
| 使用频率 | 一次性（注册即登录）。 |
| 关键痛点 | ①不知道注册是否开放（后端要求 RBAC+registration 同开才 true）；②密码规则不清导致反复被拒；③**不理解**为什么不能选租户/角色（也不该让他选）。 |
| 成功标准 | "开放时一步注册即进；未开放时入口就不该出现（fail-closed）"；"密码规则实时可见，不靠提交才知道"。 |

### 1.4 隐性画像 · 持 API Key 的直连试用者（不可回归）

现有"纯 API Key direct mode"必须继续可用（`REQUIRE_LOGIN=false` 回滚开关）。设计红线：**API Key 绝不进入管理中心**；它只服务能力试用。这个画像不新增 UI，只作为"别破坏既有"的约束。

## 2. 核心用户流（step-by-step，含岔路与错误态）

> 术语：**版本冲突** = 你基于旧 `version` 回写、服务端已变更。按 FINAL_PLAN §9.2 冻结契约：缺 `If-Match` → **428**；`version` 陈旧 → **412**（响应含 `currentVersion` + 安全字段摘要）；业务冲突（重名/角色被引用/最后管理员）→ **409**；管理写被灰度关 → **503 `rbac_writes_disabled`**。（与任务简报"冲突返回 409"的口径差异见 §6 问题 Q1，本文按 §9.2 的更精确分型设计，UI 文案层面对用户统一表达为"内容已被他人更新/该操作暂不可用"，不暴露原始码。）

### 流程 A — 管理员登录 → 进管理中心 → 检索用户 → 改角色/启停/重置密码 → 保存

**主路径**
1. 管理员在 `/login` 用账号密码登录（现有 LoginView 流程）。登录响应带 `effectiveScopes`（含 `role-admin`）。
2. 顶栏身份 chip 出现：`alice · acme · Bearer`；因含 `role-admin`，顶栏/侧栏出现**管理中心**入口（普通用户此入口不存在）。
3. 点击"管理中心" → 懒加载 `/admin/users`（默认落用户页）。首屏拉取第一页（服务端分页，默认 **50/页**，`X-Total-Count` 驱动分页器）。
4. 顶部检索/筛选：`用户名 / 租户 / 角色 / 启用状态`。输入 **300ms 防抖**，切换筛选时**取消上一在途请求**（防乱序覆盖）。
5. 命中行展示：`用户名 · 租户 · 角色(chips) · 有效scopes摘要(计数+可展开) · 启用开关 · 版本(藏在详情)`。
6. 点某行 → **右侧抽屉**打开用户详情（**保留列表上下文**，不整页跳转）：
   - `direct scopes`：**只读**，带"本期只读"标签（业务规则 §3.8）。
   - `roles`：可编辑，`RolePicker` 多选 chips。
   - **有效 scopes 实时预览**：`union(directScopes, ∪ 所选角色的 scopes)`，标注"预览 · 保存后由服务端最终确定"（客户端用 `adminRoles` 里的 `role.scopes[]` 计算，仅改善判断，不是权威）。
   - 可选：`启用/停用` 开关、`重置密码` 子操作（默认收起）。
7. 改动 `roles` 后点"保存" → 携 `If-Match=version` 回写。
8. **成功** → 抽屉内 toast「已保存」+ 列表行就地更新（新角色/新有效scopes/新 version）+ **一条常驻说明**：「权限变更在该用户**下次刷新令牌后**生效；已签发的旧令牌最长在 TTL 内仍有效」（诚实披露 §3.14；对被降权用户，后端已撤 refresh，此处告知"其当前会话将在下次续期时失效"）。

**岔路 · 启用/停用**
- 停用是破坏性：走 `DangerConfirmDialog`。
- **停用/降权的是自己或最后一个管理员** → 保存前**先本地预警**：「你正在移除自己的管理员权限，保存后将失去管理中心访问」/「这是最后一个启用的平台管理员」。用户仍可尝试；后端在事务内是权威。
- 若后端返回 **409 last_admin** → 明确文案：「系统必须保留至少一个启用的平台管理员，此操作被拒绝」+ 不清空草稿。

**岔路 · 重置密码**
- 子操作单独展开，**永不预填、永不回显**；输入即时校验密码策略（min/max 来自 public-config / admin config）。
- 提交成功 → toast「密码已重置」+ 说明「该用户其余会话将被登出（需重新登录）」（假设 A3，见 §6）。密码只入不出、不落 storage/URL/日志。

**错误态 · 版本冲突（核心可恢复路径，不丢草稿）**
- 保存返回 **412**（或缺 If-Match 的 428）→ 弹 `VersionConflictDialog`：
  - 三栏并列：`字段` · `你的草稿` · `服务端最新`，高亮**发生差异**的字段。
  - 主操作「查看最新并重做」：用服务端最新值刷新详情，但**保留你的草稿供逐字段再决定**；**不提供无脑覆盖**。
  - 次操作「放弃我的修改」。
- 关键体验保证：**任何冲突都不会让管理员丢失刚才输入的内容**。

**错误态 · 管理写被灰度关（503）**
- 全管理域进入**只读态**：所有"保存/删除/停用"按钮禁用并附说明「管理写入当前已关闭（维护/灰度）」；GET/列表仍可用于诊断。fail-closed，不让用户白填一堆再失败。

**错误态 · 会话在中途失权**
- 任一管理 API 返回 403（角色被他人撤销、access token 已换新）→ 跳 `/forbidden`，文案「你的管理员权限已变更，请重新登录以获取最新权限」，提供"重新登录"按钮。承认 access token 延迟不可即时回收。

### 流程 B — 建角色 → 选 scope → 看影响用户数 → 保存

**主路径**
1. `/admin/roles` → "新建角色"。
2. 表单：`名称`（唯一键）、`描述`、`ScopePicker`（按域分组：对话 / 知识 / 智能体 / 审批 / 分析 / 通道 / 多模态 / 平台管理；说明来自计划新增 `scopeCatalog.ts`，人话解释每个 scope）。
3. 选 scope 时右侧实时汇总"已选 N 项 scope"，`平台管理`（含 `role-admin`）分组给**加重视觉**并附提示「这是平台级高权限，谨慎授予」。
4. 新建时 `assignedUserCount=0`，无影响预览（因为还没人被分配）。
5. 保存（`POST /auth/admin/roles`，仅创建）→ 成功 toast + 回列表。

**岔路 · 编辑已有角色的 scope（"看影响用户数"在这里才有意义）**
1. 进角色详情 → 顶部常驻：「此角色当前分配给 **N 个用户**」（`assignedUserCount`）。
2. 改 scope 前后，明确「改动将影响这 N 个用户**下次获取令牌后**的权限」。
3. `N` 可点 → 跳 `/admin/users?role=<角色名>`（带筛选深链），让管理员**先去看看到底是谁**再决定。
4. 保存走 `PUT /auth/admin/roles/{name}` + `If-Match`；冲突同流程 A 的 `VersionConflictDialog`。

**错误态**
- 重名（**409**）→「角色名已存在，请换一个名称」，不清空表单。
- 未知 scope（真实角色里存在但 catalog 没有）→ **原样显示与保留**，标"未知 scope"，不丢弃（防目录漂移，§3 一致）。

---

### 流程 C — 删除在用角色被挡

1. 角色列表某行"删除" → 若 `assignedUserCount>0`：**前端先软拦**——删除按钮旁直接显示「N 个用户在用，不能删除」，点击给 `DangerConfirm` 也不放行，改为引导「先解除这 N 个用户的该角色」→ 一键跳 `/admin/users?role=<名>`。
2. 若 `assignedUserCount=0` → `DangerConfirmDialog`「确认删除角色 X？此操作不可撤销」→ 携 `If-Match` 删除。
3. 即便前端软拦，仍以后端为权威：并发下后端返回 **409 role_in_use** → 明确复述「该角色已被引用，无法删除」并刷新计数（可能有人刚分配）。
4. 删除最后一个管理员角色/导致零管理员的路径由后端 **409 last_admin** 兜底，文案同流程 A。

---

### 流程 D — 自助注册 → 即登录

1. 应用启动时 `bootstrap()` 后**尽力**拉一次 `GET /auth/public-config`（失败不阻断登录，登录页正常渲染）。
2. LoginView 仅当 `registrationEnabled===true` 时显示"没有账号？去注册"入口；否则**入口不出现**（fail-closed）。
3. `/register`（懒加载，仅 public-config 开时可达；直达而未开 → 重定向登录并提示"注册暂未开放"）：
   - 字段：`用户名`、`密码`、`确认密码`。**无 tenant / role 选择**（§3.9）。
   - 密码规则**实时**显示与校验（min/max 来自 public-config），不靠提交才报错。
4. 提交 `POST /auth/register {username,password}` → **成功即登录**（后端直接给会话）→ 跳首页。
5. 错误态：
   - 用户名已存在（409）→「该用户名已被占用」，保留已填。
   - 频率限制（429）→「注册过于频繁，请稍后再试」。
   - 注册被关（503 / 运行时开关变化）→「注册当前未开放」，隐藏/禁用提交，fail-closed。

---

### 流程 E — 普通用户被挡在管理域外（深链 → Forbidden）

1. 普通用户（无 `role-admin`）登录后：顶栏/侧栏/命令面板**均无**管理入口（同一 permission source 裁决，无旁路）。
2. 手工深链 `/admin/users` 或 `/admin/roles`：
   - 路由守卫（Bearer-only + `requiredScopes: ['role-admin']`）判定不足 → 跳 `/forbidden`。
   - `ForbiddenView` 文案：「此页面需要**平台管理员**权限（`role-admin`）。当前账号不具备。」+ 返回首页按钮。
3. **前端不是授权边界**：即便绕过 UI 直接打 `/auth/admin/**`，后端仍 403（§3.6）。UI 的 gate 只改善体验。
4. 特例：API Key 覆盖 Bearer 时访问 `/admin/**` → 因管理域 Bearer-only，视为"无有效管理身份" → Forbidden，并提示「管理中心不使用 API Key，请清除 API Key 或用管理员账号登录」。

---

### 流程 F — RAG 共享库入库与跨库查询

1. 进 RAG 工作台。仅当 `SHARED_KB_UI_ENABLED`（构建期）**且** `GET /rag/config.publicEnabled===true`（运行时）→ 顶部出现 **「当前租户库 / 共享知识库」双 tab**；否则只显示租户库并给准确说明（不宣称支持）。
2. **文档库/检索台**沿用现有双区布局，但：
   - tab 切换即切换 `visibility` 视图（`tenant` / `public`）。
   - 命中卡新增**来源 badge**：`租户` / `共享`，取自服务端 `KnowledgeHit.visibility`（**不靠 docId/名称推断**，§3 一致）。跨库查询时结果合并、按 badge 区分来源。
3. **上传入库**：显式选择 `visibility=tenant|public`。
   - 选 `public` → `DangerConfirm`「此文本将对**所有租户**可检索，确认公开？」（需 `public-ingest` scope；无则反应式 403 转人话）。
   - **共享图片本期禁用**：图片入库在 public tab 下置灰并说明「共享库暂不支持图片」（§3.12）。
4. **共享文档管理**（列表/详情/删除）：public tab 下的删除需 `public-ingest`，走 `DangerConfirm`；后端 `list/get/delete` 需支持 visibility（依赖 §8.4，属后端缺口，本文按契约设计）。
5. GraphRAG 分区维持 flag-off 诚实锁定，且**明确"图谱不并入共享分区"**（§3、§8.4 `GraphRetrievalSource` shared 恒 false），不误导用户以为共享覆盖图谱。
6. 普通用户在共享开时可见共享文档**元数据**；**是否可见完整正文 = 待验证**（§6 Q2，本文默认"元数据 only"）。

## 3. Job Stories（When… I want… so I can…）

**身份与凭证确定性**
- J1. When 我同时登录了账号又手填了 API Key，I want 一眼看清此刻哪种凭证在生效并能一键清除 Key，so I can 不会用错身份误操作（尤其误以为在用管理员身份）。
- J2. When 我作为普通用户浏览控制台，I want 管理相关入口对我完全不可见，so I can 专注在自己能用的能力上、界面不被无关项干扰。

**用户管理**
- J3. When 安全事件发生，I want 用租户/角色/状态快速筛出目标账号并一键停用，so I can 在最短路径内止血。
- J4. When 我给某用户改角色，I want 在保存前实时预览它的有效 scopes，so I can 确认授权到位、不多不少再提交。
- J5. When 另一个管理员和我同时改了同一个用户，I want 保存时被清楚告知"内容已更新"并能看到差异、保留我的草稿重做，so I can 不静默覆盖别人、也不丢自己的输入。
- J6. When 我要重置某人密码，I want 密码规则实时可见、输入永不回显，so I can 一次填对且不担心泄露。
- J7. When 我正在改动自己或最后一个管理员，I want 在保存前收到显式预警，so I can 不把自己/平台锁死。

**角色管理**
- J8. When 我新建角色选 scope，I want scope 按业务域分组并有人话解释、高权限项被显著标注，so I can 快速、准确、谨慎地组合权限。
- J9. When 我要改一个已在用角色的 scope，I want 先看到"影响 N 个用户"且能一键跳去看是谁，so I can 评估爆炸半径后再决定。
- J10. When 我试图删除一个仍被引用的角色，I want 被明确挡下并被引导去先解除引用，so I can 不破坏一批人的权限。

**注册**
- J11. When 注册开放时我是新人，I want 一步注册即进入、密码规则实时可见、不用纠结选租户/角色，so I can 零门槛开始使用。
- J12. When 注册未开放，I want 注册入口根本不出现，so I can 不做无效尝试、也不误解平台状态。

**权限边界与诚实披露**
- J13. When 我深链到无权的管理页，I want 落到清楚说明原因的 Forbidden 页，so I can 知道是权限问题而非页面坏了。
- J14. When 我改了权限，I want 被告知"下次刷新令牌才生效、旧令牌有 TTL 延迟"，so I can 对生效时机有正确预期、不反复怀疑没保存成功。
- J15. When 某能力对我锁着，I want 一句话区分"没权限(scope)"还是"没开(flag)"并给出下一步，so I can 知道该找管理员还是等开关。

**共享知识库**
- J16. When 我查询命中来自不同库，I want 每条命中都有"租户/共享"来源标识，so I can 判断信息归属、不误信来源。
- J17. When 我要把一段文本公开进共享库，I want 一次显式确认"全租户可检索"，so I can 有意识地公开、不误操作泄露。

## 4. 验收标准（可测，贴合"易用/顺手/流畅"）

> 编号 AC，均为可观测/可测的通过条件。默认在桌面主流分辨率 + 键盘/鼠标；移动与无障碍另列。

**效率 · 最短路径**
- AC1. 从"已在用户列表"到"改完某用户角色并保存成功"，**核心点击 ≤ 4 步**（选行 → 改角色 → 保存 → 确认 toast）。
- AC2. 停用一个账号（已定位到行）**≤ 2 步**（开关/停用 → 确认）。
- AC3. 用户检索输入到列表刷新，感知延迟符合 **300ms 防抖**，且快速连打不产生**乱序覆盖**（旧请求被取消）。
- AC4. 大数据量下（10k 用户）列表**服务端分页**，首屏与翻页在性能预算内，无 N+1 卡顿（依赖后端 §8.3；前端不一次性拉全量）。

**安全可恢复 · 不丢草稿（红线）**
- AC5. 版本冲突时**必然**弹冲突对话框，展示"你的草稿 vs 服务端最新"字段差异，**不存在**任何"无脑覆盖"按钮，且刷新最新后草稿仍在。
- AC6. 破坏性操作（删除/停用/降权自己或最后管理员/公开入库）**必有**一次显式确认（DangerConfirm 或内联两段式确认）。
- AC7. 管理写关闭（503）时，全域进入只读，写按钮禁用并说明原因，**不会**让用户填完再失败。

**身份清晰**
- AC8. 顶栏**始终**可见 `用户名 · 租户 · 凭证模式`；API Key 覆盖登录身份时有**高对比警告 + 清除**入口，且**管理入口消失**。
- AC9. 普通用户在顶栏、侧栏、命令面板**均无**任何管理入口；管理深链**必然**落 Forbidden 并说明原因；直接 HTTP 仍由后端 403。

**缺权可发现且说清原因**
- AC10. 任一锁定能力/页面都给出**可读原因**并区分 `scope-required` 与 `flag-off`（复用现有五态心智），且缺权项**仍可被发现**（不隐藏、可预览/复制 curl），只是不可执行。
- AC11. Bearer 已登录时，能力可用性按 `effectiveScopes` **预判**（缺 scope 提前提示）；API Key 模式保持"权限未知、反应式 403"，二者不混淆。

**诚实披露**
- AC12. 任何角色/scope 变更的成功反馈都**明示**"下次刷新令牌生效 + 旧令牌 TTL 延迟"。
- AC13. 共享命中来源 badge **只来自服务端 `visibility`**；图谱检索明确标注"不并入共享"。

**注册**
- AC14. `registrationEnabled=false`（或 public-config 拉取失败）时，注册入口**不出现**；`=true` 时一步注册**成功即登录**并进首页；用户无处可选 tenant/role。
- AC15. 密码策略（min/max）**实时**在注册与重置密码处可见并即时校验，错误在**提交前**暴露。

**无障碍 · 移动 · 韧性**
- AC16. 抽屉/对话框可**键盘操作**（Tab 焦点循环、Esc 关闭、焦点归位），破坏性确认默认焦点不落在危险按钮上。
- AC17. 管理页面在窄屏可用（列表可横向或卡片化降级，抽屉转全屏），沿用现有响应式收纳心智。
- AC18. 管理路由**懒加载**且**不被 catalog 拉取失败阻塞**（catalog 挂了，管理中心照常可用）。
- AC19. 凭证（access token / API Key / 密码）**绝不**进入 localStorage/sessionStorage/URL/日志/截图（延续现有硬约束）。

## 5. 优先级（P0/P1/P2）与最小可用切片

### 5.1 优先级矩阵

**P0 — 管理闭环 + 身份地基（安全关键，必须一起上）**
- 顶栏身份 chip（用户名/租户/凭证模式）+ API Key 覆盖高对比警告与清除；管理入口仅 Bearer role-admin 可见（AC8/AC9）。
- Bearer-only 管理壳 + `role-admin` 路由裁决 + `/forbidden`（流程 E）。
- 用户：列表（服务端分页 + `用户名/租户/角色/状态`筛选，300ms 防抖 + 取消在途）、详情抽屉、**改角色**、**启停**、保存（`If-Match`）+ **VersionConflictDialog（不丢草稿）**（流程 A 主干、AC1/AC3/AC5）。
- 角色：列表、建角色（分组 ScopePicker）、改 scope、删角色**在用保护**（流程 B/C、AC6/AC10）。
- 破坏性操作确认、写关闭只读态、诚实披露文案（AC6/AC7/AC12）。
- direct scopes 只读展示（§3.8）。

**P1 — 自助与共享（可与 P0 并行，各自独立 flag）**
- 重置密码子操作（流程 A 岔路、AC15）。
- 注册页（public-config 驱动、密码策略实时、成功即登录）（流程 D、AC14）。
- RAG 双 tab + visibility 上传/删除确认 + 命中 `租户/共享` badge（流程 F、AC13）；依赖后端 §8.4。
- 有效 scopes 实时预览（J4，纯前端增强，若后端 role.scopes 可得即可做）。
- 建/改角色时"影响 N 用户"与一键跳用户筛选（J9）。

**P2 — 顺手增强（锦上添花，不阻塞发布）**
- 命令面板（⌘K）纳入管理动作（同一 permission source，避免旁路）。
- 用户列表批量操作（批量停用/批量改角色）、保存的筛选视图。
- 有效 scopes 差异解释器（"这次改动新增/移除了哪些 scope"）。
- 可复制的诊断 `X-Trace-Id`（错误时展示，便于排障）。
- 键盘优先编辑（在列表上直接快捷键改状态）。

### 5.2 最小可用切片（Thinnest Viable Slice）建议

> 目标：用**最小改动**交付"安全的管理价值"，先让少量 bootstrap 管理员（如 alice）可用，再逐步放量（对齐 FINAL_PLAN §12.3 灰度顺序）。

**Slice 1（第一个可发布单元）= P0 中的"只读 + 身份地基"**
1. 身份 chip + API Key 覆盖警告 + 管理入口 Bearer-only 显隐 + Forbidden 深链。
2. 用户列表（分页/筛选）**只读** + 角色列表**只读**。
   → 对应灰度第 3 步"RBAC UI 开但只读，仅 bootstrap admin 验证列表"。**零写入风险**，先把身份/导航/权限门/性能验证掉。

**Slice 2 = 打开写闭环**
3. 用户改角色/启停/保存 + VersionConflictDialog；角色建/改 scope/删（在用保护）。
   → 对应灰度第 4 步"开 admin writes，小范围创建测试角色/用户，观察冲突/撤 session"。**此切片必须带冲突对话框与破坏性确认一起上，不可拆散**（否则违反红线 AC5/AC6）。

**Slice 3（可并行）= 注册 + 共享库（各自 flag，默认关）**
4. 注册页（P1）。
5. RAG 双 tab + visibility（P1，依赖后端 §8.4 就绪）。
   → 对应灰度第 5/6 步。

产品判断：**Slice 1 单独就有价值**（管理员可安全巡检 + 全站身份清晰），且几乎无写风险，是最理想的首发。**重置密码不进 Slice 2**（可 P1 补），避免首个写切片过大。

## 6. 待澄清问题 / 假设（缺失需求不臆造）

> 沿用 FINAL_PLAN 的"待验证"项并标注。凡会改变**数据可见性**或**服务 Key 权限**的问题，实施前必须向用户确认（FINAL_PLAN §14）。

**问题（需产品/后端确认，可能影响契约或数据可见性）**
- **Q1 · 版本冲突状态码口径**：任务简报写"冲突返回 409"，FINAL_PLAN §9.2/§15 明确分型为 **412（陈旧）/ 428（缺 If-Match）/ 409（业务冲突：重名/引用/最后管理员）**。本文按 §9.2 更精确分型设计错误处理（`errors.ts` 需据此扩展，§8.1 已列）。**需确认最终冻结为哪套**；对用户文案层不暴露原始码，但前端分支逻辑依赖它。
- **Q2 · 共享文档正文可见性（BR §3.13，待验证）**：普通登录用户在共享开时可见共享文档**元数据**；**能否查看完整正文**未定，当前 `DocumentInfo` 不含正文。本文默认"**元数据 only**"，共享检索命中片段（query 返回的 text）视为已授权可见。**若未来要开放正文浏览，属数据可见性变更，必须先确认。**
- **Q3 · 管理员创建用户时的 tenant 输入方式**：admin 可"建"用户（任务简报），需 `username/password/tenant/roles`（directScopes 只读，故新用户靠 roles 授权）。**无"列出租户"接口**，`tenant` 是**自由文本 + 校验（非空、≠`__public__`）** 还是从某来源下拉？本文假设**自由文本 + 前端校验**，标为假设 A1，需确认是否需要租户枚举来源。
- **Q4 · 重置密码是否登出该用户其余会话**：FINAL_PLAN 对"权限降低"要求撤 refresh；密码重置是否同样撤销该用户会话未显式声明。本文假设 A3=**是（撤销其 refresh 会话，需重新登录）**，需确认。
- **Q5 · 共享图片"禁用"的呈现**：本文假设在 public tab 下**显示但置灰 + 说明原因**（比"完全隐藏"更可发现，符合 AC10 诚实心智）。需确认是否可接受，或要求完全隐藏。
- **Q6 · 用户列表默认页大小**：FINAL_PLAN §7.3 写"默认 50"，本文采用 50。若产品希望更小首屏（如 25）请确认——影响首屏密度与翻页频率。

**假设（本文据此设计，风险较低，实施可沿用，但欢迎推翻）**
- **A1**（=Q3 默认）建用户 tenant 为自由文本 + 校验 `≠__public__`（§3.9）。
- **A2** 用户编辑采用**右侧抽屉**（保留列表上下文）而非整页；窄屏降级为全屏抽屉。这是"顺手"取舍，非契约。
- **A3**（=Q4 默认）重置密码撤销该用户其余会话。
- **A4** direct scopes **本期只读展示、不在 UI 修改**（业务规则 §3.8 已确认，非待验证——此处仅重申以免实现方误做可编辑）。
- **A5** "有效 scopes 实时预览"由前端用 `adminRoles` 的 `role.scopes[]` 计算，**仅供判断、非权威**，保存后以服务端 `effectiveScopes` 为准（避免与后端并集口径漂移的风险由"预览"措辞承担）。
- **A6** 管理中心作为**独立顶层入口**（`/admin`），**不混入**能力 SideNav 的模块分组，避免与"能力试用"心智串味；普通用户完全不渲染该入口。
- **A7** 注册页/管理页均**懒加载**且不依赖 catalog（AC18），catalog 失败不影响其可用（对齐 FINAL_PLAN §12.1"管理页面拖慢普通用户"缓解）。
- **A8** 现有 LoginView 的 `DEMO_PASSWORD='demo12345'` 自动提交属安全点，本产品设计**默认关闭 demo 一键登录**（由 `DEMO_LOGIN_ENABLED` flag 控制，生产 false），演示卡最多只预填用户名（对齐 §8.1 LoginView 改动项）。**不改代码，仅在产品层要求**。

## 7. 给下游专业 Agent 的衔接说明

- **UI/UX 与交互状态 Agent**：重点消化 §2 流程 A 的抽屉编辑 + VersionConflictDialog、§2 流程 F 双 tab、§4 无障碍/移动验收（AC16/AC17）。复用现有 `WorkbenchSection`/`InfoNote`/`EmptyState`/`DangerConfirm`（拟新增）/五态 badge 与 `btn--danger` 内联两段式确认心智（RAG 现有 `pendingDelete` 模式可升级为对话框）。
- **前端架构 Agent**：§5 的切片顺序与"P0 写闭环不可拆散冲突对话框"是硬约束；管理域 Bearer-only、懒加载、独立 store/API、不引入 API Key 与 catalog 门禁（对齐 FINAL_PLAN §6.1、§8.2）。
- **仓库约束/可复用组件 Agent**：优先复用 `authStore/sessionStore/gate/errors/layout`，`credentialMode` 与 `usePermission` 是新增的单一裁决点，须替换所有 `executionGate(...)` 旧调用（§8.1 列了全部调用点）。
- **测试/风险/边界 Agent**：§4 的 AC5/AC6/AC8/AC9 是必测红线；§6 的 Q1/Q2 未决前，冲突码分支与共享正文可见性测试需按"假设"打桩并标注。
