# 设计模式梳理

> 本文梳理 `langchain4j-platform`（Java 21 + Spring Boot 3.3.5 的 DDD 全微服务 AI 平台）中**实际落地**的设计模式。每一条都对照真实代码给出代表类与位置，**没有代码支撑的模式一律不写**，并在末尾诚实列出"排查过但未使用"的模式。
>
> 路径均相对仓库根，采用 `模块/.../类名.java` 简写；行号为撰写时快照，可能随代码演进漂移，**以类名/方法名为准**。
>
> 阅读建议：先看「一、三条设计主线」，它们贯穿全局、后文大量引用；再按需查「GoF / 微服务 / 框架 / AI 编排」四类详解；最后「速查矩阵」可当索引。

## 一、三条设计主线（先读）

整个平台的模式选择围绕三条主线展开，理解它们就理解了 80% 的代码结构：

### 主线 1：接口 + `@ConditionalOnProperty` 多实现（可插拔 + 本地零依赖）

存储 / 传输 / provider 几乎都是「**一个接口 + 内存/Noop 默认实现 + 由环境变量属性开启的 Http/Kafka/Redis/Jdbc 变体**」。全仓 `@ConditionalOnProperty` 约 **211 处**、`matchIfMissing = true` 约 **61 处**。

- 消费方只依赖接口，换后端零改业务代码 → 本质是 **策略模式（Strategy）** 的装配期变体，也是 **特性开关（Feature Toggle）**。
- 默认全部走内存/确定性实现（如 knowledge 的 `HashEmbeddingModel` 不真调 embedding），**本地运行与单测无需任何外部基础设施**。
- 支持灰度三态（如 `app.rag.authz.mode` = disabled/shadow/enforce）。

代表：`EmbeddingStore`/`EmbeddingModel`（`KnowledgeEmbeddingConfig`）、`GraphStore`、`AsyncTaskStore`、`Reranker`、`GroundingChecker`、`EventPublisher`、`RateLimiterRegistry`、`ChannelEventPublisher` …

### 主线 2：全局唯一 `ChatModel` Bean + `ChatModelListener` SPI（横切关注点旁挂）

- **所有 LLM 调用**经 `GatewayChatModelFactory` 构建、指向外部 LiteLLM 单一 OpenAI 兼容端点 —— provider 路由/failover/模型名映射下沉到 LiteLLM，**Java 侧没有任何 provider `switch`**。
- 进程内**刻意只注册一个 `ChatModel` Bean**（`@ConditionalOnMissingBean`），以贴合 langchain4j `@AiService` 按类型自动发现的约束；级联等变体绝不注册成第二个 Bean。
- 审计 / 成本 / token 预算 / OTel 追踪都实现为 `ChatModelListener`，由 Spring 收集成 `List` 灌进模型 —— **横切逻辑与业务/LLM 调用路径彻底解耦**（观察者/SPI）。

### 主线 3：租户 + traceId 全链路透传（多租户隔离随每一跳传播）

- 请求身份放 ThreadLocal `TenantContext`（**上下文对象模式**）；跨服务 REST 调用由出站拦截器把身份重新编成短时内部 JWT（`X-Internal-Token`）+ traceId 注入，下游 filter 校验后还原进 ThreadLocal。
- 边缘网关校验外部凭证后统一换发内部 JWT，**下游只信内部 JWT**（网关卸载 / 令牌换发）。
- 业务代码零感知，测试须在 `@AfterEach` 中 `TenantContext.clear()` 防串租。

## 三、GoF 经典设计模式

### 创建型

#### 工厂（Factory）
把「构造复杂对象」的逻辑集中到工厂，屏蔽底层细节。

- `platform-gateway-client/.../GatewayChatModelFactory.java` —— `build(modelName, temperature)` / `buildDeterministic()`（temp=0 判官变体）/ `buildStreaming()`；工厂持有配置与全部 `ChatModelListener`，构造模型时统一 `.listeners(...)` 灌入，保证同步/流式/级联三条链路的 token/成本/审计口径一致。
- `knowledge-service/.../DocumentSplitterFactory.java` —— `create()` 内 `switch(strategy)` 在 recursive / markdown-header / parent-child / semantic 间选择并组装切分器。
- `platform-gateway-client/.../cascade/CascadeChatModelFactory.java` —— 组装 `CascadeChatModel`。

**动机**：取代原单体的 provider `switch`（选择下沉 LiteLLM）；集中 listener 注入点。

#### 建造者（Builder）
几乎所有多参可选对象都用流式 `.builder()...build()`。

- langchain4j 声明式服务：`agent-service/.../AgentConfig.java` 中 `AiServices.builder(AgentBrain.class).chatModel(...).build()`（DAG planner/critic/replanner 等 11 处）；`analytics-service/.../Nl2SqlConfig.java` 用 `.tools(SqlQueryTool)` 单独挂工具而不污染普通对话。
- 模型构造 `OpenAiChatModel.builder()`；HTTP 客户端 `RestTemplateBuilder.rootUri().additionalInterceptors(...).build()`。

**动机**：可选装配项（工具、记忆、内容注入器、listener）组合多，「只挂需要的」既可读又可单测。

#### 单例 + 依赖注入（Singleton via Spring + DI）
全局共享对象一律做成 Spring 单例 Bean 由容器注入，最典型是**全局唯一 `ChatModel` Bean**。

- `platform-gateway-client/.../PlatformGatewayClientAutoConfiguration.java` —— `@Bean @ConditionalOnMissingBean(ChatModel.class)`，刻意的单例守卫（langchain4j `@AiService` 按类型枚举 `ChatModel`，多于 1 个即冲突）。
- 协作者一律**构造器注入**（非字段 `@Autowired`），可选依赖用 `ObjectProvider<T>` 或 `@Autowired(required=false)` setter（而非被本项目明令禁用的 `@ConditionalOnBean`）。

**动机**：贴合 langchain4j 自动发现约束；横切 listener 通过 `List<T>` 注入自动挂载；无 Spring context 的极速单测。

### 结构型

#### 适配器（Adapter）
一批 `Http*Client` 把「远端服务的 HTTP 端点」适配成本地跨服务接口，调用方只依赖接口、不感知 HTTP。

- `agent-service/.../client/HttpKnowledgeClient.java`（`implements KnowledgeClient`，内部 `restTemplate.postForObject("/rag/query", ...)`）；同族约 15 个：`HttpAnalyticsClient`、`HttpWorkflowClient`、`interop` 的 `HttpAgentInteropClient`、`channel/voice` 的 `HttpConversationClient`、`eval` 的 `HttpRetrievalClient` …

**机制**：把 DTO 适配为一次带租户/traceId 透传的 REST 调用，并把 `RestClientException` **降级**为空回复（不外抛）。默认 `@ConditionalOnProperty(matchIfMissing=true)` 装配。

#### 装饰器（Decorator）
「实现同一接口 + 持有 delegate + 增强后转发」，透明叠加能力。

- `platform-gateway-client/.../cascade/CascadeChatModel.java`（`implements ChatModel`，包裹 cheap/strong 两个 `ChatModel`，低置信才升级）—— 因它实现 `ChatModel`，任何吃 `ChatModel` 的地方（`AiServices.builder`）都透明用上级联。
- `knowledge-service/.../PrefixingEmbeddingModel.java`（给 delegate 加 `search_query:` / `search_document:` 非对称前缀后转发，只影响送 embed 的副本）。
- 切分器 `ParentChildSplitter` / `MarkdownHeaderSplitter` / `SemanticChunkingSplitter` 均持有 fallback/child delegate `DocumentSplitter`。

#### 代理（Proxy，动态代理）
langchain4j `@AiService` 接口在运行时被生成为**动态代理**，把「方法调用 + `@SystemMessage`/`@V`/`@MemoryId` 注解」翻译成一次带模板渲染与记忆管理的 `ChatModel` 调用。

- `conversation-service/.../Assistant.java`（`@AiService` 接口，`chat(...)` 无实现体）、`StreamingAssistant`、`agent-service/.../AgentBrain.java`。接口本身没有实现类，代理在 `invoke` 时渲染系统提示模板（`@V` 占位）、按 `@MemoryId` 取会话记忆、调用注入的 `ChatModel`。

**动机**：业务只声明意图接口，**LLM 始终藏在可 mock 接口后**，controller/service 单测可直接注入接口 mock。

#### 外观（Facade）
每个下游服务的 `@RestController` 对外充当子系统外观；服务内部再由聚合服务把多个协作者收敛成一个入口。

- `agent-service/.../DeepAgentService.java` 把可插拔工具注册表、ReAct 决策核心、观察拼接封装在一个方法后，对外只暴露 `/agent/run`。
- 各服务 `@RestController`（`/chat`、`/rag/**`、`/agent/**`）作为对内部装配/工具/编排的统一门面。

### 行为型

#### 策略（Strategy）—— 本项目主导写法
同一契约多实现，运行时或装配时择一（见**主线 1**）。两种择取时机：

- **运行时按数据/配置选**：`knowledge-service/.../search/FusionStrategy.java`（枚举 `WEIGHTED_MAX`/`RRF`，`effectiveDefault()` 依 ES 是否参与查询动态择取）；`DocumentSplitterFactory` 的 `switch`。
- **装配时经「接口 + `@ConditionalOnProperty`」择一**：`Reranker`（Noop/Llm/Jina 三选一，默认 Noop）、`GroundingChecker`（Noop/Llm）、`EmbeddingStore`/`GraphStore`/`ChannelMessageDispatcher`/`RateLimiterRegistry`/`EventPublisher` …

#### 观察者 / 监听器（Observer / Listener）
主体在关键生命周期点回调一组注册的监听器，与主体解耦。

- **`ChatModelListener` SPI**（本仓库明确的横切约定）：`GatewayChatModelFactory` 构造时收 `List<ChatModelListener>` 灌进模型，每次 LLM 调用触发 `onRequest/onResponse/onError`。实现：`platform-audit/.../AuditChatModelListener.java`（审计）、`platform-metering` 的成本 / token 预算 listener、`platform-observability/.../otel/OtelChatModelListener.java`（追踪）。
- **`AgentRunListener`**：`DeepAgentService` 在 run 结束后对 `instanceof AgentRunListener` 的动作逐个触发 `onRunEnd()`，释放单次运行资源。

**动机**：审计/计量/成本/追踪从 LLM 调用路径彻底剥离，加一个 listener bean 即全局生效、per-tenant 归因不侵入业务。

#### 责任链（Chain of Responsibility）—— 双处强实证
请求沿一串各自「处理或放行」的处理器传递，顺序由 order 决定。

- **入站鉴权 filter 链**（edge-gateway，Spring Cloud Gateway，按 `getOrder()` 串接）：`CasdoorTokenExchangeFilter`（-120）→ `SessionBearerAuthFilter`（-110）→ `ApiKeyToInternalTokenFilter`（-100）→ `EdgeRateLimitFilter`。三者都 `implements GlobalFilter, Ordered`，对本环无法处理的请求 `chain.filter(...)` 交下一环（经典 handle-or-pass）。
- **出站 `RestTemplate` 拦截器链**：`platform-security/.../OutboundTenantForwarder.java` + `platform-observability/.../OutboundTraceForwarder.java`（均 `implements ClientHttpRequestInterceptor`），经 `AgentConfig` 的 `additionalInterceptors(...)` 组成有序链，注入内部 JWT/traceId 后 `execution.execute(...)` 交下一环。

> 诚实标注：conversation 的 guardrail→RAG→grounding→PII 是 controller 内**顺序管线**，无 handler 间 pass-along 语义，严格说归**策略/管线**更准确，不算标准责任链。

#### 命令（Command）
Agent 工具被物化为自包含可执行对象，注册进表，由调用者按名动态触发。

- 契约 `agent-service/.../AgentAction.java`（`name()`/`description()`/`run(input)`）；invoker `DeepAgentService` 持 `Map<String,AgentAction>`，`dispatch()` 里 `actions.get(name).run(input)` 执行。ReAct 大脑只产出「动作名 + 入参」，与具体 receiver（`RagSearchAction`/`AnalyticsSqlAction`/`CodeExecAction`）解耦。

#### 状态（State，枚举状态机）
任务生命周期用枚举建模，状态本身约束合法迁移与副作用触发。

- `platform-protocol/.../asynctask/AsyncTaskStatus.java`、`agent-service/.../async/AgentTaskStatus.java`：`PENDING→RUNNING→SUCCEEDED/FAILED/CANCELLED` + `isTerminal()`。终态门控 controller 拒绝再操作、`AsyncTaskStore` 释放租约、`AsyncTaskSseService` 关闭 SSE。

> 诚实标注：这是**枚举驱动的状态机**（状态决定允许的迁移与副作用），**没有** GoF State 的「每状态一个多态类」结构，算轻量体现。

## 四、微服务 / 企业集成 / 云原生架构模式

#### API 网关 + 令牌换发（API Gateway + Gateway Offloading / Token Exchange）
`edge-gateway`（Spring Cloud Gateway/WebFlux :8080）是唯一对外入口，一张路由表（`application.yml` routes，`Path=` predicate → `uri: ${SERVICE_URI}` 环境变量）把 12 个下游服务反代出去，并统一挂 CORS。

边缘校验三类入站凭证（Casdoor SSO Bearer / 会话 Bearer / `X-Api-Key`），验过后统一 mint 一枚**短时内部 JWT** 注入 `X-Internal-Token`、剥掉原始外部凭证；下游 `platform-security/.../InternalTokenAuthFilter.java` 只信内部 JWT。三条 filter 幂等协作（已有内部头即放行）。

**动机**：外部身份体系与内网信任解耦，下游零感知认证方式，密钥不泄入内网。

#### 环境上下文透传（Ambient Context Propagation）
见**主线 3**。`RestTemplate` 装 `OutboundTenantForwarder` + `OutboundTraceForwarder` 两个拦截器，出站编内部 JWT + traceId；下游 `InternalTokenAuthFilter` set 进 `TenantContext` + MDC，请求末尾 `finally` clear 防串租；anonymous 不带头。

#### 事务性发件箱（Transactional Outbox）—— 三套独立实现
状态终态写与「对外通知」落在**同一 DB 事务**，异步 relay 再投递，消除「提交后崩溃丢通知」窗口，无需 2PC。

- async-task webhook outbox（HTTP 投递）：`async-task-service/.../AsyncTaskWebhookOutbox.java`（`enqueue`/`claimDue`/`markDelivered/Retry/Dead`）+ `...OutboxDispatcher.java`。
- async-task 生命周期 outbox（Kafka relay）：`AsyncTaskLifecycleOutbox` + `AsyncTaskLifecycleRelay`，由 `JdbcAsyncTaskStore.update()` 在终态更新的同一事务内原子写入。
- workflow 终态 outbox（Kafka relay）：`workflow-service/.../WorkflowTerminalEventOutbox.java`，建在 Flowable **同一** `workflowDataSource`，由 BPMN `end` 事件的 `ExecutionListener` 在引擎事务内 `enqueue`，与 `ACT_*`/`WF_REPLY` 原子提交。

relay `@Scheduled` 轮询 `claimDue(now)` → 成功 `markDelivered`，失败指数退避 `markRetry`，超上限进 `DEAD` 死信。

#### 事件驱动 / 发布订阅 + 幂等消费（Event-Driven Pub-Sub + Idempotent Consumer）
- `platform-eventbus` 抽象 `EventPublisher`，默认 `NoopEventPublisher`（零 Kafka 依赖），`platform.eventbus.enabled=true` 且 classpath 有 Kafka 时切 `KafkaEventPublisher`（三条件 `@ConditionalOnMissingBean`+`OnClass`+`OnProperty` 择一）。发送以 `key=tenantId` 保序，**同步阻塞至 broker ack** 才返回（供 outbox「先确认落 broker 再 markDelivered」）。
- 消费侧用 `ProcessedEventStore`（内存默认 / `=jdbc`）按稳定 `eventId`（如 `workflow:<instanceId>`）**去重**：先查→处理→成功后 `markProcessed`；处理抛异常则不标记 → 重投再处理（不丢），已完成事件重投被跳过（去重）。

**= outbox 至少一次 + 消费去重 → 实用 exactly-once**，无需 Kafka 原生事务。

#### 竞争消费者 + 退避重试 + 死信（Competing Consumers / Retry-Backoff / DLQ）
- webhook outbox 支持多实例并发派发，每实例用唯一 `ownerId` **claim-TTL 租约**认领行（乐观并发：`UPDATE ... WHERE STATUS='PENDING' OR (IN_PROGRESS AND CLAIMED_UNTIL<=now)`），claim 过期可重认领 → 水平扩容且互不重复、故障自愈。
- 应用层指数退避（`delay = base * 3^n`）：4xx 直接死信、5xx/网络错误重试至上限进 `DEAD`（可 `listDead` 巡检）。Kafka 层 `DefaultErrorHandler` + `DeadLetterPublishingRecoverer` + `FixedBackOff`，耗尽投 `<topic>.DLT`。
- 生产者幂等：`enable.idempotence=true, acks=all`（`KafkaProducerConfig`），并为未来端到端事务预留 `transactional-id-prefix`（默认不启用）。

#### 降级与安全默认（Graceful Degradation / Fail-closed）
- **降级**：`conversation-service/.../HttpKnowledgeClient.java` catch `RestClientException` → 返回空 `KnowledgeQueryReply`，对话照常继续（而非 500）；整个 RAG 客户端本身 feature-flag，未开则装空实现。
- **Fail-closed / Fail-open 授权**：文档级 ReBAC（委托外部 auth-platform/SpiceDB）依赖故障时按 mode 分流 —— `RealKnowledgeAuthz` 在 `SHADOW` 放行全集（fail-open + 打点）、`ENFORCE` 拒绝全部（fail-closed）；读用 `Consistency.fullyConsistent()` 保证刚授权即生效。

#### 编排式工作流 + 人在环（Orchestration + Human-in-the-Loop）
- Flowable 退款审批 BPMN：`processes/refund-approval.bpmn20.xml`（`assess → 网关 → 人工 UserTask 审批 → 网关 → resolve/reject → end`），ServiceTask 调 Spring bean。`WorkflowService` 同步执行，返回时要么 `COMPLETED` 要么停在 `WAITING_APPROVAL`；async executor 刻意关闭；租户隔离靠流程变量 `processVariableValueEquals("tenantId", …)`。
- **定时对账 / 超时清扫**：`ApprovalTimeoutSweeper`（`@Scheduled` 扫挂起超时的审批 → 自动驳回，规避 BPMN boundary timer 在 async executor 下的 ThreadLocal 真空坑）。
- 多 Agent 侧的编排见「六、AI 编排」的 DAG。

#### 健康探针 + 外部化配置（Health Probes / Externalized Config）
- 全模块引 `spring-boot-starter-actuator` 暴露 liveness/readiness；Helm 配 `livenessProbe`/`readinessProbe`；compose 用 `depends_on: condition: service_healthy` 门控启动次序。
- 各服务 `spring.config.import: "optional:configserver:..."` —— Config Server 不可达不阻断启动，`${ENV:default}` 兜底（配置中心是增强非硬依赖）。

## 五、框架级 / 数据访问 / 契约模式

#### SPI 自动装配（Spring Boot AutoConfiguration.imports）
6 个共享库 `platform-*` 无主类，靠 Boot 3 的 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` **自注册** —— 服务只要依赖该库并配好属性即自动获得其 bean，无需 `@Import`。含 gateway-client / security(2) / metering(2) / eventbus / observability(2) / audit。横切能力（`ChatModelListener`）亦走此 SPI。

#### DTO / 值对象（Value Object，不可变 record）
所有跨服务 DTO 集中于 `platform-protocol`（约 **70 个** `public record`），不可变，作为服务间**单一事实源**（禁止各服务重复定义）。

- 如 `AsyncTask.java` 用**紧凑构造器做防御性拷贝**（`Map.copyOf` / `List.copyOf`）与 null 归一，并保留向后兼容的重载构造（新增字段不破坏旧构造点）。`TenantContext` 内嵌 `record Tenant(tenantId,userId,scopes,department)` 带 `hasScope()` 行为方法。

#### 仓储 / DAO（Repository/DAO）—— 裸 JdbcTemplate + DDL 字面量演进
无 JPA/MyBatis/Flyway；`Jdbc*Store` 用裸 `JdbcTemplate` 直连 MySQL，**表结构靠代码里的 `CREATE TABLE IF NOT EXISTS` / `ALTER TABLE ADD COLUMN` 字符串演进**（改表 = 改这些代码块，非加迁移文件）。

- `async-task-service/.../JdbcAsyncTaskStore.java`（构造器 `init()` 建表 + `addColumnIfMissing()` 吞重复列异常增列；`update()` 用 `TransactionTemplate` 把读-改-写 + 生命周期 outbox 写在同一事务）。同族 15+：`auth` 的 `Jdbc{UserAccount,Group,Role,RefreshSession,...}Store`、`knowledge` 的 `JdbcGraphStore`、`platform-eventbus` 的 `JdbcProcessedEventStore`。默认内存实现使本地零 DB。

#### 上下文对象（Context Object，ThreadLocal）
见**主线 3**。`TenantContext`（`ThreadLocal<Tenant>`，`current()` 未设返回 `ANONYMOUS` 兜底，`captureRaw()` 供跨线程拷贝，`clear()`）。同族：`CategoryContext`、`RetrievedSourcesContext`、`SqlExecutionContext`。ThreadLocal 不跨网络 —— 靠内部 JWT 在下游重建。

#### 空对象 / Noop + 内存默认（Null Object / In-Memory Default）
可选协作者的默认实现是「什么都不做」的 Noop 或纯内存实现，真实实现按开关注入 —— 消除判空分支，默认关闭的功能零配置可跑。

- Noop（14+）：`NoopReranker`、`NoopQueryExpander`、`NoopKnowledgeAuthz`、`NoopGroundingChecker`、`NoopEventPublisher` …
- InMemory：`InMemoryTokenBudgetTracker`、`InMemoryRateLimiterRegistry`、`InMemoryGraphStore`、`auth` 的 `InMemory*Store` …

#### 注册表 / 提供者（Registry / Provider）
- **注册表**：`List<T>` 收集所有实现，构造时装进 `Map<name,T>` 供运行时按名分派。`AgentAction` 表（`DeepAgentService` 按 `name()` 注册、ReAct 按名 `actions.get(...)` 分派，新增工具 = 加一个带开关的 bean）、`InteropToolRegistry`（MCP 工具目录）、`RateLimiterRegistry`、`DocumentRegistry`。
- **提供者**：`provider` 属性选后端 —— `EmbeddingStore`/`EmbeddingModel`（默认 `InMemoryEmbeddingStore` + 确定性 `HashEmbeddingModel`）、`VisionModel`/`DefaultVisionModel`、`SchemaProvider`。

#### 配置属性绑定（@ConfigurationProperties）
35+ 个类型安全属性绑定（前缀 `platform.*` / `app.*`），字段内联默认值即「未配即默认」，部分实现 `InitializingBean` 做启动期 fail-fast 校验（如 `RagAuthzProperties`）。每个环境变量在 README 有对应 curl。

## 六、AI / LLM 编排模式

#### ReAct 深度 Agent（ReAct Deep Agent）
思考→选动作→观察的自主循环，每步 LLM 只结构化决策下一步。

- `agent-service/.../AgentBrain.java`（`@SystemMessage` 明写「按 ReAct 一次只决定下一步」的决策核心接口）+ `DeepAgentService.java`（执行引擎）+ `AgentDecision.java`（`thought/action/actionInput/note/finalAnswer` 结构化输出，字段带 `@Description` 供 LLM 抽取）。
- 内建多重停机：`MAX_STEPS / TIMEOUT / BUDGET(token 预算) / LOOP(打转检测,签名去重) / CANCELLED / ERROR`；scratchpad 超限经 `ScratchpadSummarizer` 压缩；`delegate` 动作按 `maxDepth` 递归派子 Agent。

#### 工具调用注册表（Tool / Function Calling Registry）
见「命令模式」。统一 `AgentAction` 契约，`description()` 即工具说明书直接进提示，`List<AgentAction>` 自动构成可插拔工具表；全部 `@ConditionalOnProperty` 门控（`code_exec`/`mcp_call` 双门控默认关）。`McpToolAction` 把 langchain4j MCP 的 `ToolSpecification` 动态桥接进本地注册表。

#### 多 Agent DAG 编排（Orchestrator-Workers / Planner-Executor）
LLM 规划子任务 DAG → 拓扑分层 → 同层并行跑独立子 Agent → 综合，含可选评审-重规划闭环。

- `agent-service/.../dag/AgentDagService.java`（`topologicalLevels` 无环分层 + `CompletableFuture` 并行 + `synthesisGoal` 综合）+ `AgentDagPlanner`（拆 1~6 子任务）+ `AgentDagCritic`（打分）+ `AgentDagReplanner`（低分改计划重跑，`maxReplans` 限次）。进度经 `AgentTaskProgressSink` emit。`DataAnalystPlanner`/`ProcessPlanner`（人在环）复用同一 DAG 引擎。

#### 提示链 / 路由 / 投票 / 反思（Prompt Chaining / Routing / Ensemble / Reflexion）
- **提示链** `chaining/PromptChainService.java`：输入依次过固定步骤，每步一次 LLM 调用，步间**确定性 gate**（长度/包含/正则）不过即短路。流程写死、不由模型决定 → 可重复可单测。
- **路由** `conversation-service/.../routing/QueryRouterService.java`：LLM-as-Router 三段式 classify→dispatch→answer，把 query 分到 RAG/TOOL/CHAT，仅 RAG 分支走检索注入；分类器用 temp=0 判官模型（`QueryClassifier`），异常降级 RAG。
- **投票** `voting/VotingService.java`：同问并行 N 票，`majority`（归一化计票 + `minAgreement` 判 `confident`，适合离散题）或 `synthesis`（temp=0 `VoteAggregator.merge` 收敛，适合自由文本题）。
- **反思 / LLM-as-judge** `reflexion/ReflexionService.java`：answer→critique→improve 循环至加权分达阈值或用尽 `maxAttempts`；判官 `AgentDagCritic`（正确性/完整性/清晰度三维 0~1）为 Reflexion 与 DAG replan 共用。判官/自评的确定性靠 `GatewayChatModelFactory.buildDeterministic()`（temp=0），用于 `VoteAggregator`/`QueryClassifier`/`ConfidenceGate`/`LlmReranker` 等收口。

#### 检索增强生成（RAG）
每轮**新鲜注入** `<source>` 来源块、多源融合、LLM 重排、查询扩展、上下文分块、图谱召回，全部 feature-flag、默认确定性/内存实现。

- `conversation-service/.../RagPromptAugmenter.java`：检索命中拼 `<source id=...>` 经 `@V("context")` 注入系统提示，**来源不落多轮记忆**（避免累积膨胀），同时回带结构化 hits 供 grounding 核对。
- `knowledge-service/.../search/HybridFusionService.java`：`WEIGHTED_MAX`/`RRF` 两策略融合向量/关键词/ES/图谱四源（`1/(k+rank)`，免疫量纲差）。
- `rerank/LlmReranker.java`（逐候选 0~1 相关性打分 + minScore 截断）、`query/LlmQueryExpander.java`（改写 N 个近义变体多路召回）、`ingest/LlmContextualEnricher.java`（Anthropic Contextual Retrieval：chunk 前拼文档级上下文再嵌入）、`graph/GraphSearchService.java`（GraphRAG：实体链接 → 多跳邻居三元组）。

#### 模型级联回退（Model Cascade）
见「装饰器」。`cascade/CascadeChatModel.java`：cheap 先答，命中工具调用直接返回不升级；否则 `ConfidenceGate`（长度 + `uncertainty-markers` + 可选便宜模型 temp=0 自评）判低置信才 `strong.chat`。cheap/strong 各挂全部 listener 照常计量，且不注册成第二个全局 ChatModel Bean。

#### 护栏 + 事实核对（Guardrail + Grounding）
- `conversation-service/.../guardrail/ConversationGuardrail.java`：前置注入护栏（`block/sanitize/audit` 三模式）+ 后置 PII 输出脱敏，纯确定性、流式/非流式一致。
- `grounding/LlmGroundingChecker.java`：Layer0 确定性查伪造 `[doc=ID]` 引用 + Layer1 `FaithfulnessScorer`（RAGAS 风格 0~1 支撑度）打分不足 → 追加可信度告警后缀（warn 模式不拒答，不改正文）。护栏/grounding 均**默认开**（仅按需切换模式收紧）。

#### 语义缓存 + 记忆（Semantic Cache / Memory）
- **L1 语义缓存** `cache/SemanticCache.java`：pre-RAG、问题级、按租户分桶的向量相似缓存，`getOrCompute` embed 问题→租户桶 `findNearest`→余弦 ≥ 阈值命中即短路后续 embedding/检索/LLM，否则跑 supplier 回填；支持定向失效。默认开。
- **记忆**：短期 `memory/ChatMemoryConfig.java`（`ChatMemoryProvider` 三档 window-mode：messages/tokens/summary，按 `@MemoryId` 会话隔离，summary 模式溢出即 LLM 摘要）；长期 `memory/profile/`（`ProfileExtractor` temp=0 增量抽取用户事实，`recall` 作为 `@V("context")` 新鲜注入、每轮 `observe` 更新，画像不落多轮记忆）。

## 七、明确排查过但未使用 / 不适用的模式（诚实清单）

| 模式 | 结论 | 说明 |
|---|---|---|
| 原型 Prototype | 未使用 | 无 `clone()`/`Cloneable`/原型注册表；工厂每次 `build()` 全新构造。 |
| 组合 Composite | 未使用 | `AgentAction` 是注册表按名选一，多 Agent 用 DAG 图，非同接口递归树。 |
| 桥接 Bridge | 未使用 | 「接口 + 多 `@ConditionalOnProperty` 实现」属策略，非抽象/实现双维度独立继承。 |
| 享元 Flyweight | 未使用 | 无共享细粒度对象池；单例 Bean 属 DI 非享元。 |
| 模板方法 Template Method | 未使用 | main 代码**无任何 `abstract class`**，无「抽象骨架 + 子类填空」。 |
| 迭代器 Iterator | 未使用 | 无自定义 `Iterator`/`Iterable`，均用 JDK 集合。 |
| 中介者/备忘录/访问者/解释器 | 未使用 | 无专门实证。 |
| 断路器 Circuit Breaker | 未使用（自研替代） | 无 Resilience4j/Hystrix；弹性靠 outbox 退避 + Kafka `DefaultErrorHandler` + fallback。 |
| 边车 Sidecar | 不适用 | 上下文透传靠**进程内拦截器**，非独立 sidecar 容器。 |
| Kafka 原生事务 exactly-once | 未启用 | `transactional-id-prefix` 骨架在但默认空，走「幂等生产 + outbox + 消费去重」路线。 |

> 另注：CLAUDE.md 曾提到的 `ImageTextProvider` 接口在当前代码已被 `VisionModel`/`DefaultVisionModel` 取代。
