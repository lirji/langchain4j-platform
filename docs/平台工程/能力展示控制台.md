# 能力展示与试用控制台（前后端分离）

一个面向开发者的控制台,用于**展示并直接试用**平台各服务能力。**前后端分离**:前端是独立静态工程 `capability-showcase-frontend/`(Vue3 + TS + Vite),不属于后端 Maven 构建,可独立部署到任意静态托管(nginx / Netlify / CDN / 对象存储)。

> 能力→模块拆分建议见 [能力前端模块拆分建议](能力前端模块拆分建议.md)。前端工程自述见 `capability-showcase-frontend/README.md`。

## 架构（前后端分离）

- **前端**:纯静态 SPA,浏览器以 **direct mode** 带用户输入的 `X-Api-Key` **跨域直调** `edge-gateway` 暴露的业务能力(`/chat`、`/rag/query`、`/agent/run` …)。
- **能力目录**:默认打包为静态 `catalog.json`(由 `capability-showcase-frontend/capabilities.yml` 生成),**零后端依赖**;也可 `VITE_CATALOG_URL` 指向后端动态目录。
- **后端改动(direct-mode 基线)**:`edge-gateway` 开启 CORS(`GATEWAY_CORS_ORIGINS` 白名单 + 允许 `X-Api-Key`/`Authorization`/`If-Match` 头 + 暴露 `X-Trace-Id`)。**仍无展示后端服务、不改任何业务 controller/DTO/表/消息结构。**
- **登录与授权(增量、默认关、引入即安全)**:除 direct-mode 手输 `X-Api-Key`,前端还支持**账号密码登录**(`auth-service` 会话 JWT)+ **RBAC 管理控制台**(`/admin`,持 `role-admin` 可见),并可接 **Casdoor OIDC 多租户 SSO**(方案C:Shared Application + 登录前选组织,每租户以 `<base>-org-<tenant>` 客户端登录)。后端侧 `edge-gateway` 相应新增 **Casdoor token 交换路径**(`CasdoorTokenExchangeFilter`,`EDGE_CASDOOR_ENABLED=false` 默认关;开启后验 Casdoor token 换发内部 JWT,下游零改动),知识库另有文档级细粒度授权(`RAG_AUTHZ_MODE=disabled` 默认关)。故「后端唯一改动=CORS」仅对 direct-mode 基线成立。详见 [公网化 OIDC 改造方案](公网化-OIDC-改造方案.md) 与前端 `README.md`。
- **API Key** 仅存浏览器内存,绝不落 URL/localStorage/日志。

## 运行

**本地开发**(前端 dev + 本地网关):
```bash
cd capability-showcase-frontend
npm install && npm run dev        # http://localhost:5173,业务请求经 vite 代理到 :8080 网关(同源无跨域)
```

**整套 docker-compose**(网关 + 各服务 + 独立前端 nginx):
```bash
docker compose -f deploy/docker-compose.yml up --build
# 前端 http://localhost:8093,网关 http://localhost:8080(已配 GATEWAY_CORS_ORIGINS 放行 :8093)
```

**独立构建/部署**(跨域):
```bash
cd capability-showcase-frontend
VITE_EDGE_BASE_URL=https://api.example.com npm run build   # 产物 dist/,扔任意静态托管
```
跨域部署务必在网关 `GATEWAY_CORS_ORIGINS` 加入前端来源。

## 使用要点

- **API Key**:顶栏输入网关认可的 `X-Api-Key`(见 `edge-gateway` `application.yml` 的 api-keys)。加载目录不需要,试用业务能力需要。
- **能力五态**:`ready` 可执行;`ready-degraded` 降级实现(如 `/rag/query` 默认 HashEmbedding 内存库);`flag-off` 需开 `app.xxx.enabled`;`scope-required` 需 `ingest`/`approve` scope 否则 403;`display-only` 危险端点默认不可执行(如 `DELETE /workflow/data`)。
- **三形态**:JSON、multipart 上传、SSE(浏览器 `fetch`+`ReadableStream` 消费逐 token 流)。

## 配置（前端 VITE_*，见 capability-showcase-frontend/.env.example）

| 变量 | 默认 | 说明 |
|---|---|---|
| `VITE_EDGE_BASE_URL` | 空(同源/dev 代理) | 业务网关基址;跨域分离部署必填 |
| `VITE_BASE` | `/` | SPA 部署子路径 |
| `VITE_CATALOG_URL` | 内置 `catalog.json` | 改为后端动态目录接口 |
| `VITE_LIVE_DISCOVERY` | `true` | 是否启用 live discovery |
| `VITE_AUTH_MODE` | `apikey` | 认证模式:`apikey`(默认)/`oidc`/`dual`(Casdoor OIDC 灰度开关) |
| `VITE_REQUIRE_LOGIN` | `true` | 强制登录守卫;`false` 退回纯 API Key 老流程(回滚开关) |
| `VITE_RBAC_CONSOLE_ENABLED` | `true` | `/admin` 管理控制台入口(灰度/回滚;真正可用仍需 `role-admin`) |
| `VITE_SHARED_KB_UI_ENABLED` | `true` | 共享库双视图(仍取决于服务端 `/rag/config.publicEnabled`) |
| `VITE_CASDOOR_ISSUER` | `http://localhost:8000` | Casdoor issuer(oidc-client-ts authority,自动 discovery) |
| `VITE_CASDOOR_CLIENT_ID` | `ragshared0client00000001` | Casdoor **Shared Application** base client_id(方案C;每租户 `<base>-org-<tenant>`) |
| `VITE_DEMO_LOGIN_ENABLED` / `VITE_DEMO_PASSWORD` | `true` / 空 | 演示账号一键登录卡(生产置 `false`);口令仅内部/本地注入,**生产绝不设置** |

> Casdoor 回调/登出/静默续期路径与 scope 等完整 `VITE_CASDOOR_*` 见前端 `capability-showcase-frontend/README.md` 与 `.env.example`。

网关侧:`GATEWAY_CORS_ORIGINS`(逗号分隔的允许来源白名单,**勿用 `*`**);Casdoor 边缘开关 `EDGE_CASDOOR_ENABLED`(默认 `false`)/`EDGE_CASDOOR_MODE`(`dual`(默认)|`only`)/`CASDOOR_AUDIENCES`(默认 `ragshared0client00000001`,按 `<base>-org-*` audience 家族 + `(owner,aud)` 绑定校验)。

## 能力覆盖

9 个模块、82 条能力全部落地(以 `public/catalog.json` 生成结果为准,对照真实 controller 核验):Chat/RAG/Agent/Async/Analytics(含各自 flag-off 高级能力) + Workflow/Multimodal/Interop-Eval/Channel。后四者有领域专用工作台视图(`ModuleHost.SPECIALIZED`:审批串联、图像/语音分区、MCP/A2A+评测、出站锁定+回调 header 注入),其余走数据驱动的 `GenericModuleView`;均复用共享 `CapabilityRunner` 与执行闸门(flag-off/scope/危险确认),不绕过安全裁决。支持 `multipart-sse`(如 `/voice/chat/stream`) 与 `in:header` 参数注入(不覆盖 `X-Api-Key`);`/eval/gate` 的 HTTP 422 作为业务门禁结果展示。

## 能力开关（`deploy/docker-compose.yml` 默认状态）

整套 compose 现已把「前端页面涉及的能力」尽量默认打开。三档：

**① 默认全开（内置 DeepSeek 栈 + 栈内服务即可跑通，前端 catalog=`ready`）**
- 对话：`/chat/auto` 意图路由、`/chat/cascade` 级联降级、`/chat/memory` + `/memory/profile` 长期画像
- 知识库：GraphRAG `/rag/graph/**`；数据分析：NL2SQL `/chat/sql`
- Agent：业务流程智能体 `/agent/process/**`（→ 栈内 workflow-service）

**② 默认开、装配 boot-safe，但实际调用需自备外部资源（catalog 也标 `ready`，缺资源时调用期报错，非启动崩溃）**
- 语音 `/voice/**`：需 OpenAI 兼容 ASR/TTS。设 `VOICE_API_KEY`，或把 `VOICE_BASE_URL` 指向本地 whisper/tts 网关。
- 图片 RAG `/rag/image`、`/rag/image-search`：需多模态 embedding 端点。配 `RAG_MULTIMODAL_BASE_URL`（默认 `http://localhost:8000/v1`）、`RAG_MULTIMODAL_MODEL`（默认 `jinaai/jina-clip-v2`）、`RAG_MULTIMODAL_API_KEY`。

**③ 默认关（开启前必须先备好外部资源，否则服务启动即崩）——catalog 仍 `flag-off`**
- 图像描述 / 图文对话（vision）：`/vision/caption`、`/chat/vision`。两步：
  1. 在 `deploy/litellm/config.yaml` 加一个可路由的多模态模型（如 `gpt-4o-mini` / `qwen2.5-vl`）。
  2. 置 `VISION_ENABLED=true` 且 `VISION_MODEL=<该逻辑模型名>`；图文对话再加 `CONVERSATION_VISION_ENABLED=true`。
  - ⚠️ 只置 `VISION_ENABLED=true` 而 `VISION_MODEL` 留空 → vision-service **fail-fast 崩溃**。
- MCP 增强对话（`chat.mcp`）：`/chat/mcp`。需一个可达的 MCP server（streamable-http）。置 `CONVERSATION_MCP_ENABLED=true` 且 `CONVERSATION_MCP_HTTP_URL=<你的 MCP 地址>`。
  - ⚠️ 开启但 server 不可达 → conversation-service **启动握手失败崩溃**（会一并拖垮基础对话）。

启用 ③ 后，把 `capabilities.yml` 里对应能力的 `state: flag-off` 改为 `ready`（前端执行闸门据此放行），再 `npm run gen:catalog`。

## 能力目录维护

`capability-showcase-frontend/capabilities.yml` 是目录唯一事实源(能力的方法/路径/参数/scope/flag/风险/五态)。`npm run gen:catalog`(dev/build 前自动执行)转为 `public/catalog.json`。新增/调整能力改这个文件即可。复杂/嵌套请求体以 `type:json` 原样 JSON 字段承载。

## 回滚

- 前端独立,直接下线其静态站点即可,不影响后端。
- 后端如需回退:移除 `edge-gateway` 的 `globalcors` 配置(仅此一处)。
- 因不改业务接口/表/消息结构,无需任何数据或消息迁移。
