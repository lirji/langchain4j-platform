# 长任务处理指南

本文档说明平台如何处理长耗时任务：`async-task-service`（:8086）通用任务中心的设计与用法、
各服务的接入现状，并给出分布式系统长任务通用模式的对照与当前限制。
与 [事件总线与终态可靠投递（EOS）指南](eventbus-guide.md) 互补——本文讲「任务的登记、执行与通知」，
EOS 指南讲「终态事件跨系统投递的可靠性」。

> 一句话总览：长任务的本质是把「一次调用」变成「一条有状态的记录」，然后围绕这条记录解决三个问题——
> **谁来执行**（租约）、**怎么通知**（推拉结合）、**坏了怎么办**（幂等 + outbox + 死信）。
>
> **核心设计：async-task-service 不执行任务。** 它是一个基于租约（lease）的任务登记/分发中心，
> 执行永远由外部 worker（如 agent-service）完成——这让任务中心保持通用、无业务逻辑。

---

## 1. 任务生命周期

### 1.1 任务记录与状态机

跨服务契约在 `platform-protocol`（`com.lrj.platform.protocol.asynctask`）：

- `AsyncTask`：14 字段不可变 record——`taskId, tenantId, userId, kind, status, input(Map), result, error, webhookUrl, createdAt, updatedAt, finishedAt, leaseOwnerId, leaseExpiresAt`；
- `AsyncTaskStatus`：`PENDING / RUNNING / SUCCEEDED / FAILED / CANCELLED`，`isTerminal()` = 后三者。

```
PENDING ──lease──▶ RUNNING ──PATCH status──▶ SUCCEEDED / FAILED
   │                  │
   └────DELETE────────┴──────▶ CANCELLED          （终态更新写 finishedAt 并清空租约；终态后幂等 no-op）
```

### 1.2 REST 端点

入口 `async-task-service/src/main/java/com/lrj/platform/asynctask/AsyncTaskController.java`，
全部按 `TenantContext` 租户隔离，每次状态变更发 `AsyncTaskEvent` 应用事件 + 写审计。

| 端点 | 方法 | 说明 |
|---|---|---|
| `POST /async/tasks` | 提交 | 建 `PENDING` 任务返回 202；缺 `kind` → 400；客户端可自带 `taskId` 作**幂等键**，重复提交 → 409 |
| `GET /async/tasks` / `/{taskId}` | 查询 | 按当前租户列举 / 单查 |
| `POST /async/tasks/{taskId}/lease` | 领取 | worker 带 `workerId` + `leaseSeconds`（默认 60，clamp 1–3600）原子抢占 → `RUNNING` |
| `PATCH /async/tasks/{taskId}/status` | 回报 | 校验租约持有者（`workerId` 必须匹配）；终态是幂等 no-op |
| `DELETE /async/tasks/{taskId}` | 取消 | 仅非终态可取消 → `CANCELLED` |
| `GET /async/tasks/{taskId}/stream` | SSE | 状态流，支持 `Last-Event-ID` 断点续传 |
| `GET /async/webhook-outbox/dead` | 巡检 | webhook 死信列表 |

## 2. 执行模型：租约（lease）而非内置执行器

worker 的工作循环是 `POST lease` 领取 → 执行业务 → `PATCH status` 回报。可靠性来自租约语义：

- **原子抢占**：JDBC 下 lease 是条件 `UPDATE`（状态非终态 + 租约为空/自己/已过期），多 worker 竞争安全（`JdbcAsyncTaskStore.lease`）；
- **崩溃自愈**：worker 挂掉后不需要任何清理动作——租约到期（`leaseExpiresAt`）任务自动可被其它 worker 重新领取；
- **回报防串**：`updateStatus` 校验 `leaseOwnedBy`，过期租约或他人租约的回报被拒绝。

服务内唯一线程池 `asyncTaskExecutor`（`AsyncTaskWebConfig`，core 2 / max 8 / queue 100）**只用于内存模式的
webhook 异步投递**，不跑任务。

**租户上下文传播**：线程池挂 `TaskDecorator`——提交线程 `TenantContext.captureRaw()` + 复制 MDC，
工作线程 `set` / finally 还原（agent-service 的 `AgentAsyncConfig.agentTaskExecutor()` 同一套模式）；
跨服务 REST 调用由 `OutboundTenantForwarder` + `OutboundTraceForwarder` 拦截器透传租户与 traceId。

## 3. 进度与通知：三条通道

### 3.1 SSE 拉流（`AsyncTaskSseService`)

`GET /async/tasks/{taskId}/stream`。监听 `AsyncTaskEvent`，每次状态变更向该 taskId 的所有 `SseEmitter`
推**整份任务快照**；维护有界历史（64 条）支持 `Last-Event-ID` 断线续传；终态自动 `complete` 关流。
注意粒度：任务中心只推状态快照，**细粒度执行进度**（步骤/百分比）在执行方实现——见 agent-service 的
`AgentTaskProgressSink`（`agent-service/src/main/java/com/lrj/platform/agent/async/`）。

### 3.2 Webhook 推送（终态回调）

提交任务时带 `webhookUrl`，终态时任务中心主动回调。两套实现按存储模式切换：

| 模式 | 实现 | 可靠性 |
|---|---|---|
| JDBC | `AsyncTaskWebhookOutboxEnqueuer` 监听终态事件入 `ASYNC_TASK_WEBHOOK_OUTBOX` 表（`OUTBOX_ID=taskId` 一任务一行）→ `AsyncTaskWebhookOutboxDispatcher` 定时（默认 30s）`claimDue` 抢占（claim TTL，支持多实例）→ HTTP POST | **4xx → 直接死信**；5xx/网络错误按 `base × 3^(attempts-1)` 指数退避，默认 3 次后进 `DEAD`；死信可经 `GET /async/webhook-outbox/dead` 巡检 |
| 内存 | `AsyncTaskWebhookNotifier` 在 `asyncTaskExecutor` 上直投 + 退避重试 | best-effort，重启即丢 |

回调头：`X-Async-Task-Id`、`X-Async-Task-Status`、`X-Tenant-Id`。

### 3.3 Kafka 生命周期事件（`transport=kafka`）

终态发 `AsyncTaskLifecycleMessage` 到 topic `platform.asynctask.lifecycle`（key=tenantId，
`eventId=asynctask:<taskId>:<status>`）。JDBC 模式下走**事务性 outbox**：`JdbcAsyncTaskStore.update`
在终态更新的**同一事务内**原子写 `ASYNC_TASK_LIFECYCLE_OUTBOX`，`AsyncTaskLifecycleRelay` 定时 relay 到
Kafka；内存模式退化为 `AsyncTaskKafkaNotifier` 提交后 best-effort 直发。消费侧（channel-service）按
eventId 去重，整条链路 = effective exactly-once——机制详解见 [EOS 指南](eventbus-guide.md)。

## 4. 持久化

存储遵循平台惯例「接口 + 内存默认 + 属性开启的 JDBC 变体」：

- `AsyncTaskStore`：基类兼内存实现（`ConcurrentHashMap`），代码注解默认（`matchIfMissing=true`）；
  `withStatus` / `withLease` / `leaseAvailableFor` 为两实现共享的纯函数；
- `JdbcAsyncTaskStore`（`store=jdbc`）：裸 `JdbcTemplate` 管 `ASYNC_TASK` 表，
  `CREATE TABLE IF NOT EXISTS` + `addColumnIfMissing` 演进表结构（本仓无迁移文件的持久化惯例），
  数据源/事务由 `AsyncTaskJdbcConfig` 按需装配。

> **实际默认是 JDBC**：虽然代码注解层面 in-memory 是默认，但 `application.yml` 与
> `deploy/docker-compose.yml` 的缺省值都是 `${ASYNC_TASK_STORE:jdbc}`，实际运行默认落 MySQL。
> JDBC 模式持久化全部字段（含 in-flight 的 `PENDING/RUNNING`）；重启后 in-flight 任务依赖
> 「租约过期 → worker 重新 lease」恢复，无主动扫描（见 §6 限制）。终态任务由 `cleanup` 定时按
> TTL（默认 24h）清理。

## 5. 各服务接入现状

| 服务 | 角色 | 机制 |
|---|---|---|
| **agent-service** | worker / 镜像方 | 门控 `app.agent.async.external.enabled`（默认关）。`ExternalAsyncTaskClient` 调 `/async/tasks/**`（kind=`agent.task`）。两种模式（`ExternalAsyncTaskProperties`）：**中央权威**——任务中心为权威存储，agent 作为 worker lease 后执行；**本地权威**——agent 本地执行（`AgentAsyncTaskService`），`AgentAsyncTaskMirror` 监听 `AgentTaskEvent` 把状态镜像到中央 |
| **workflow-service** | 提交方 | `WorkflowAsyncTaskNotifier`：审批流终态时以 `workflow-<instanceId>` 为幂等键 `POST`（**把 409 当幂等成功**），再 `PATCH` 成 `SUCCEEDED` 带结果与 `webhookUrl`，让任务中心代发 webhook。自身另有 `WorkflowOutbox`（`WF_OUTBOX` 表）做终态回推的可靠投递 |
| **channel-service** | 消费方 | `AsyncTaskLifecycleKafkaListener` 消费生命周期事件 → `ChannelCallbackService` 转外部渠道回调，`ProcessedEventStore` 按 eventId 去重；等价 HTTP 入口 `POST /channel/callbacks/async-task` |
| **knowledge-service** | 未接入 | RAG 摄取 / GraphRAG 构建 / Obsidian 导入均为服务内同步处理，未走任务中心 |

## 6. 通用长任务模式对照与当前限制

分布式系统长任务的标准解法可归成五层，平台覆盖度如下：

| 层 | 通用模式 | 平台现状 |
|---|---|---|
| 异步化 | 同步接口立即返回 202 + 任务资源，客户端凭 id 跟踪 | ✅ `POST /async/tasks` → 202 |
| 结果获取 | 轮询（兜底）/ SSE·WebSocket（低延迟）/ webhook（主动推）三者并存 | ✅ 三条通道齐全（§3） |
| 可靠执行 | 租约 + 过期重领（防 worker 崩溃卡死）、提交幂等键、事务性 outbox | ✅ 三件齐全，是平台做得最标准的部分 |
| 失败与恢复 | 超时看门狗、有限重试 + 死信、checkpoint 断点续跑、saga 补偿 | ⚠️ 部分——webhook 层有重试+死信；任务层缺看门狗（见下） |
| 可观测 | 全链路 traceId、审计、租户配额 | ✅ `platform-observability` + `platform-audit` + `platform-metering` |

### 限制与后续演进

以下能力**当前不存在**（按文档维护原则，列作演进方向而非现状）：

- **任务级超时看门狗**：无扫描器把超时的 `RUNNING` 任务置 `FAILED` 或重新入队；租约过期仅允许被动重领，若无 worker 再来 lease，任务将停留在 `RUNNING`；
- **重启后 in-flight 主动恢复**：JDBC 下 in-flight 任务留库但无重投循环，同样依赖 worker 主动 lease；
- **任务级重试端点**：重试只存在于 webhook / lifecycle outbox 投递层，任务本身失败后需重新提交；
- **webhook 签名**：回调无 HMAC 签名，接收方无法验真伪（当前仅内网互调，公网化前需补）；
- **knowledge-service 摄取接入任务中心**：大批量导入目前同步阻塞，是天然的接入候选。

## 附录：开发协作中的长任务

长任务的心法在系统层与工程协作层是同构的——**状态外置、步骤幂等、随时可恢复**。
用 Claude Code / AI 辅助做跨会话的长开发任务时的对应做法：

1. **计划先行、拆成可验收的阶段**：先出实施计划（如 `docs/plans/` 下的计划文档），每阶段有明确验收标准，做完一段验证一段；
2. **进度落盘，不依赖上下文记忆**：维护进度文件（`PROGRESS.md` / 计划文档内清单）；上下文被压缩后以进度文件为准——文件比对话可靠，等价于任务记录 + checkpoint；
3. **小步提交**：本仓的 Conventional Commits 让 git 历史本身成为进度记录，等价于幂等的状态回报——会话丢失后 `git log` + 进度文件即可无损接续；
4. **子代理分摊**：探索 / review / 批量修改派给子代理只回收结论，等价于 lease 出去的 worker。
