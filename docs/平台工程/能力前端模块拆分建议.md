# 能力 → 可拆分独立前端模块 建议

> 回答"平台哪些能力可以拆成独立前端模块"。基于独立前端工程 `capability-showcase-frontend/` 的能力目录（`capabilities.yml` → 静态 `catalog.json`），端点均对照真实 controller 核验。
>
> **实现状态（2026-07-11）**：下述 9 个模块已在前端全部落地（共 80 条能力）。P0 五模块 + 四个 P1/P2 模块均可用；`workflow/multimodal/interop-eval/channel` 有领域专用工作台视图，其余走数据驱动通用视图。本文的优先级/边界分析仍是"如何进一步拆成独立可部署 micro-frontend"的指导。

## 拆分原则

1. **按业务域（≈微服务边界）切分**，不按技术形态切——一个模块 = 一个心智单元，与后端服务基本对齐。
2. **每个模块可独立开发/独立路由/独立演进**：物理上首期是"单 SPA 多路由"，但代码按 `modules/<domain>/` 隔离，共享一个 App Shell；direct mode 下各模块只依赖 App Shell、彼此无运行时耦合，因此**天然可拆成独立 micro-frontend**。
3. **App Shell（地基）不是业务模块**：API Key 会话、catalog 加载、JSON/multipart/SSE 统一客户端、能力五态徽章、任务时间线组件、traceId 展示——所有模块的公共底座，必须最先做。

## 能力五态徽章（"诚实呈现"的产品骨架）

平台一半能力默认关闭（feature-flag），若 UI 把它们画成"可用"就是误导。故每条能力标注五态之一：

| 状态 | 含义 |
|---|---|
| `ready` | 默认可执行（如 `POST /chat`） |
| `ready-degraded` | 可执行但为内存/确定性降级（如 `/rag/query` 默认 HashEmbedding，非真实语义；接真实向量库需配 qdrant/nomic） |
| `flag-off` | 需开 feature flag 才注册（nl2sql/vision/voice/cascade/mcp/router/graph 等），默认不可执行 |
| `scope-required` | 需 `ingest`/`approve` scope，无则 403 |
| `display-only` | 危险写/删（如 `DELETE /workflow/data`），默认仅展示不执行 |

## 模块总表

| 模块 | 对应服务 | 优先级 | 可独立拆出 | 主要交互形态 |
|---|---|---|---|---|
| **App Shell**（地基，不可拆） | — | P0 | — | 会话/目录/客户端/徽章/时间线 |
| **Chat Console** | conversation | **P0** | 高 | JSON / SSE / multipart |
| **RAG Workspace** | knowledge | **P0** | 高 | multipart / JSON |
| **Agent Lab** | agent | **P0** | 高（跨域枢纽） | JSON / SSE / 异步 |
| **Async Monitor** | async-task | **P0** | 最高（横切，供给可复用组件） | JSON / SSE |
| **Analytics Lab** | analytics | **P0**（降级） | 高 | JSON |
| **Workflow Desk** | workflow | **P1** | 中（依赖 approve scope） | JSON |
| **Multimodal Console** | vision + voice + chat/vision + rag/image | **P1** | 中（几乎全 flag-off + 音视频采集） | multipart / 音频 / SSE |
| **Interop & Eval Tools** | interop + eval | **P2** | 高（协议向，Eval 可自证 RAG） | JSON-RPC / SSE |
| **Channel Console** | channel | **P2** | 高（真实外部副作用，强二次确认） | JSON |

## 模块边界与拆分性说明

- **Chat Console**（P0，拆分性高）：`/chat`、`/chat/stream`、`/extract` 为骨干；`/chat/auto`·`/chat/cascade`·`/chat/mcp`·`/chat/memory` 为 flag-off 高级能力。纯 conversation-service，最适合作为第一个独立 micro-frontend 样板。
- **RAG Workspace**（P0，高）：`/rag/documents`（上传/列表/删）+ `/rag/query`（`ready-degraded`）为骨干；图片/GraphRAG/Obsidian 为 flag/scope 进阶。检索质量受向量库配置影响，务必用降级徽章说明。
- **Agent Lab**（P0，高，跨域枢纽）：`/agent/run`(+async)、`/agent/tasks/{id}/stream`(SSE)、DAG/链/投票/反思。注意它有独立于 async-task 的自有任务体系（`/agent/tasks/*`）。`analyst`→Analytics、`process`→Workflow 两个子能力按端点归属留在本模块，拆分时勿拆散。
- **Async Monitor**（P0，横切性最高）：`/async/tasks` 全生命周期 + SSE 断点续订 + 死信。纯工具型无业务耦合；其"任务时间线/SSE 订阅"组件应抽到 App Shell 供 Agent/Workflow 复用。
- **Analytics Lab**（P0 降级）：`/analytics/schema/tables` 默认可用；`/chat/sql` NL2SQL 默认 `flag-off`（`app.nl2sql.enabled`）。首期"活"的是 schema 浏览。
- **Workflow Desk**（P1）：退款审批 BPMN；多数操作需 `approve` scope；`DELETE /workflow/data` 为 `display-only` 默认锁定。
- **Multimodal Console**（P1）：几乎全 flag-off 且需图像/音频采集组件，前端最重，投入产出比在一期最低。
- **Interop & Eval Tools**（P2）：面向协议消费者与质量守门人，受众窄；Eval 可作为控制台"自证"工具（`/eval/retrieval` 验证 RAG）。
- **Channel Console**（P2）：有真实外部副作用（真发消息），需最谨慎的二次确认。

## MVP 与演进

- **首期 MVP = App Shell + Chat / RAG / Agent / Async / Analytics 五个 P0 模块**，覆盖 5 大交互形态（JSON / SSE / multipart / 异步状态机 / 只读浏览）。flag-off 高级能力首期"列徽章但不做交互表单"。
- **二期**：Workflow Desk、Multimodal Console，及各 P0 模块的 flag-off 能力补全表单。
- **三期**：Interop & Eval Tools、Channel Console；（可选）`CapabilityProxyController` 后端代理仅在跨源部署或统一脱敏需求出现时启用。

## 关键洞察

1. **App Shell 是组件供给方**：任务时间线、SSE 订阅、状态徽章应最早沉淀到共享层。
2. **Agent Lab 是跨域枢纽**：拆分按端点归属，别把 analyst/process 拆散。
3. **direct mode 让各模块天然可独立拆出**：只依赖 App Shell、无运行时耦合——这是各模块"可独立拆出"普遍评高的根因。
